# Recommended KiCad Architecture for Shadow Board, Operation Journal, Checkpoint Rollback, Validation, and Final Accept

## Executive conclusion

KiCadŌĆÖs native architecture strongly supports a **hybrid session model** for an AI-native PCB editor: an **authoritative shadow board** for semantics and validation, a **typed atomic operation journal** for audit, replay, and rollback, a **native KIGFX preview scene** for what the user sees, and a **single final `BOARD_COMMIT` push** when the user accepts. ## What KiCad already gives you natively

KiCadŌĆÖs rendering stack is already organized around `KIGFX::VIEW`, `VIEW_ITEM`, `VIEW_GROUP`, and `VIEW_OVERLAY`. `KIGFX::VIEW` exposes several render targets, including `TARGET_OVERLAY` and `TARGET_TEMP`, and supports `Add`, `Remove`, `Query`, visibility control, preview initialization, `AddToPreview`, and `ShowPreview`. That is exactly the shape you want for an AI-integrated display layer: the canvas already knows how to render non-destructive preview content separately from the persistent board scene. The same view also supports spatial query of visible items, which is important if you want hover, click, or hit-testing on AI preview objects rather than just passive drawing. ŅłĆciteŅłéturn13view0Ņłéturn12view0Ņłéturn12view1Ņłéturn13view2Ņłéturn11view4Ņłü

`VIEW_OVERLAY` is useful, but it is important to understand what it is. In source terms, it is a command-buffer-like preview item that stores drawing commands such as lines, circles, rectangles, polygons, stroke/fill/color/width changes, and then executes them through the GAL. That makes it excellent for transient guides, measurement hints, debug drawings, and rule-warning visualization. However, because it is fundamentally a vector of drawing commands rather than a rich board-semantic object graph, it is not the best sole representation for AI-created preview vias, tracks, and zones if you also want per-object inspection, metadata, hover cards, or fine-grained selection. For those use cases, the better pattern is to represent AI preview geometry as real `VIEW_ITEM`-style preview objects on a dedicated AI preview layer, while reserving `VIEW_OVERLAY` for lightweight annotations such as constraints, warning halos, measurement arcs, or agent intent hints. That is partly a direct reading of the rendering primitives and partly an inference from how `VIEW::Query` and item-based preview are structured. ŅłĆciteŅłéturn12view3Ņłéturn12view4Ņłéturn12view5Ņłéturn12view6Ņłéturn12view7Ņłéturn12view8Ņłéturn12view0Ņłéturn11view4Ņłü

On the model side, `BOARD_COMMIT` already behaves like the commit primitive you need for final promotion. Its `Revert()` path restores staged changes by cloning prior images and swapping item data back, updating connectivity, rebuilding selection, and clearing the staged commit state. ## Shadow board versus journal-only preview

The strongest argument **for** a complete shadow board is not aesthetics; it is **semantic completeness**. A journal plus a preview scene can show proposed geometry, but it cannot truthfully answer ŌĆ£what is the board after step 7?ŌĆØ unless something is actually applying those changes into a board model that can resolve UUIDs, nets, connectivity, zones, fill islands, and design rules the same way the live editor does. Since `BOARD_COMMIT`, `CONNECTIVITY_DATA`, `ZONE_FILLER`, and `DRC_ENGINE` all expect a board-centric world, any architecture that wants real intermediate validation, agent self-correction, and a ŌĆ£look at the result, rollback, tweak parameters, continueŌĆØ loop should maintain a real shadow `BOARD` as the authoritative state of the AI session. ŅłĆciteŅłéturn17view2Ņłéturn37view1Ņłéturn38view7Ņłéturn39view0Ņłéturn23view0Ņłü

The strongest argument **against** na├»ve shadow-board cloning is also in the source. `BOARD` deletes copy construction and assignment, maintains runtime caches, connectivity, per-item UUID maps, design settings, rule caches, and project-related references, and exposes UUID rebinding and duplicate-UUID repair helpers. That makes it unlikely that a simple in-memory copy strategy would remain correct and maintainable across refactors. In other words, ŌĆ£use a full shadow boardŌĆØ is correct, but ŌĆ£clone it with `BOARD shadow = *live;`ŌĆØ is not a viable design. The shadow board should be created either through a dedicated reconstruction path from file/serialization or through a controlled clone utility that rebuilds caches and connectivity intentionally. ŅłĆciteŅłéturn10view0Ņłéturn10view1Ņłéturn10view3Ņłéturn10view4Ņłéturn10view6Ņłéturn9view1Ņłéturn9view8Ņłéturn9view9Ņłü

Because of that, the best long-term design is a **hybrid of an authoritative shadow board and a typed journal**. The shadow board provides semantics, validation, and preview truth. The journal provides deterministic replay, rollback, auditing, and accept-time promotion. Checkpoints should therefore not mean ŌĆ£copy the whole shadow board every step.ŌĆØ They should mean ŌĆ£record a durable replay position and enough state to restore efficiently.ŌĆØ In practice, that means keeping one long-lived shadow board, journaling every atomic mutation, and using **sparse checkpoints** composed of at least a journal index, a session epoch, a handle-generation map, and either a periodic snapshot or a guaranteed replay path from the last snapshot. This is an architectural inference, but it follows directly from the fact that KiCad likes transactions and staged commits, while `BOARD` itself is not designed for trivial structural copying. ŅłĆciteŅłéturn10view0Ņłéturn17view2Ņłéturn28view0Ņłéturn30view1Ņłü

A pure ŌĆ£journal + preview scene + local validation onlyŌĆØ design is attractive for MVP simplicity, but it will break down as soon as the agent has to rely on **real refill, real clearances, real copper connectivity, or rule-aware geometry correction**. Even if you could approximate these one by one, you would be rebuilding a second semantics stack next to KiCadŌĆÖs existing board engine. For an AI-native editor embedded into KiCad itself, that is the wrong long-term bet. The durable abstraction should be: **shadow board is the source of semantic truth; journal is the source of replay truth; preview scene is the source of visual truth.** ŅłĆciteŅłéturn37view1Ņłéturn39view0Ņłéturn23view0Ņłéturn39view5Ņłü

## Operation journal, handles, checkpoints, and rollback

The operation journal should be treated as a **first-class artifact**, not just debug output. Each journal entry should minimally record: session id, step id, operation id, parent operation or script cell id, source agent, atomic op name, typed arguments, resolved board object references, created temporary handles, validation outputs, warning list, and before/after hashes or epochs for the affected shadow state. This specific field list is a recommendation rather than a KiwiCad source artifact, but it is the natural extension of how `BOARD_COMMIT` already distinguishes staged change types and how the stable IPC API moves toward explicit commit objects and undo grouping. ŅłĆciteŅłéturn17view2Ņłéturn28view0Ņłéturn30view1Ņłü

The source makes it especially clear that **temporary agent handles and KiCad UUIDs should not be conflated**. KiCad uses `KIID`-based resolution, supports rebinding item UUIDs, exposes duplicate-UUID repair, and `BOARD_ITEM` can reset its UUID. That means your AI session should maintain its own opaque `SessionHandle` namespace, with a mapping from handle ŌåÆ current shadow object identity ŌåÆ current KIID if applicable. When a rollback invalidates created objects or rewinds the board epoch, stale handles should become invalid immediately and force the runtime to re-query by handle or semantic selector. That avoids the classic ŌĆ£Python kept a pointer to something that no longer existsŌĆØ problem and makes replay deterministic even if UUID repair or object recreation changes underlying identities. ŅłĆciteŅłéturn10view4Ņłéturn9view3Ņłéturn10view6Ņłü

The most robust checkpoint model is therefore **epoch-based**. At the end of each step, the session records a new epoch, the terminal journal index for that step, and the live mapping of handles that survived. Rollback to a checkpoint means the shadow board returns to that epoch, the preview scene switches to the corresponding preview state, and any handles created after that epoch are invalidated. If a step fails part-way through, the whole step should roll back. This mirrors both CAD precedent and KiCadŌĆÖs own staged-commit mentality better than per-operation partial persistence. `BOARD_COMMIT::Revert()` is especially important here because it demonstrates that staged board mutations are already understood in KiCad as reversible grouped units, not as isolated irreversible writes. ŅłĆciteŅłéturn17view0Ņłéturn19view0Ņłéturn19view4Ņłéturn20view0Ņłü

For storage, I would not recommend snapshotting the full board after every step. Instead, use **adaptive sparse snapshots**: for example, every N steps, after expensive operations like zone refill, or when journal replay cost crosses a threshold. In between those sparse snapshots, replay the typed journal to reconstruct state. This is the same trade-off that many editors and game engines make between snapshots and command replay, and it is especially appropriate here because KiCad already has strong command/commit semantics but a non-trivial board object with large derived caches. This is still a design recommendation, but it is the one most aligned with the source constraints. ŅłĆciteŅłéturn10view0Ņłéturn17view2Ņłéturn37view1Ņłéturn39view0Ņłü

## Validation, refill, and preview feedback

Validation should be explicitly layered. The first layer is **typed-operation validation** inside KiSurf itself: unit ranges, allowed layer sets, net existence, shape degeneracy, bounding-box sanity, and obvious board-invariant checks. The second layer is **incremental board validation** after a step: recompute only the connectivity, local ratsnest, and stale regions impacted by the step, and update preview warnings immediately. The third layer is **selective refill and DRC-lite** for operations that change copper topology or rule-area interactions. The fourth layer is **full DRC** before accept, on explicit request, or when the agent asks for a high-confidence verification pass. This ladder is supported by the fact that connectivity, ratsnest, island maps, zone filling, and DRC are all separable subsystems in KiCad rather than one monolithic ŌĆ£validate everything alwaysŌĆØ call. ŅłĆciteŅłéturn37view1Ņłéturn37view8Ņłéturn37view9Ņłéturn37view10Ņłéturn39view0Ņłéturn38view2Ņłéturn39view4Ņłü

`DRC_ENGINE` already gives you the right hooks to make validation both agent-readable and human-readable. It can be bound to a board and design settings, run tests, emit violations through a handler, report progress, and attach a debug overlay. That suggests a clean implementation pattern: convert every run into both a structured `ValidationReport` object for the agent and a visual overlay for the user. The overlay can draw clash halos, isolated-island markers, rule-area shading, and hot spots. The structured report can expose stable machine fields such as item ids, rule ids, severity, location, and suggested remediation. The coexistence of a violation handler and a debug overlay in the source is a strong hint that KiCad already expects DRC outcomes to be consumable both programmatically and visually. ŅłĆciteŅłéturn38view2Ņłéturn38view3Ņłéturn39view0Ņłéturn39view4Ņłéturn39view5Ņłü

Zone refill deserves special treatment. `ZONE_FILLER` is powerful, but the source and issue tracker both show that it is a complex and somewhat fragile subsystem, especially when used through Python or in headless-like contexts. The source comment in `ZONE_FILLER::Fill()` explicitly notes a crash path for Python/API consumers when no initialized DRC engine is present, and KiCadŌĆÖs issue tracker documents refill crashes on complex designs. That does not mean you should avoid refill in the AI path; it means you should **treat it as a controlled validation phase** with progress reporting, cancellation support, and a clear distinction between ŌĆ£preview-lite without refillŌĆØ and ŌĆ£validated preview with refill.ŌĆØ For interactive AI steps, that usually means refill only when copper semantics actually matter to the next decision. ŅłĆciteŅłéturn23view0Ņłéturn23view2Ņłéturn21search6Ņłü

On the visual side, the source suggests a two-track preview strategy. Use **item-based AI preview objects** for things the user may inspect individually, and use **overlay commands** for ephemeral annotations and warnings. Store `step_id`, `op_id`, `session_handle`, and validation status in every preview object so the timeline can scrub between steps and the UI can answer ŌĆ£which script step produced this via ring?ŌĆØ That metadata is your addition, but it fits cleanly on top of KiCadŌĆÖs `VIEW_ITEM`/`VIEW_GROUP`/`VIEW_OVERLAY` infrastructure. ## Accept, reject, replay, and stale-preview handling

Final accept should be implemented as **typed-journal replay into the live board inside one `BOARD_COMMIT`**, not as ad hoc mutation of the live board during preview and not as a best-effort geometry diff from the rendered overlay. The reason is architectural: the journal is your audited, deterministic, semantically typed editing program, while `BOARD_COMMIT` is KiCadŌĆÖs native mechanism for grouping changes into a single undoable action. You already have evidence from KiCadŌĆÖs own IPC API and Python bindings that ŌĆ£begin commit now, delay visible changes, then push one grouped commitŌĆØ is the preferred stable public workflow. Your internal AI accept path should mimic that same principle. ŅłĆciteŅłéturn28view0Ņłéturn30view1Ņłéturn17view2Ņłéturn16view6Ņłü

Before replay, the session should verify a **base board hash or epoch**. If the live board has changed since the shadow session started, the accept button should not silently merge. Instead, the system should either reject the accept as stale or attempt a controlled rebase by replaying the journal onto a fresh shadow board built from the current live board and re-running validation. This stale-preview guard is an implementation recommendation, but it follows from KiCadŌĆÖs synchronous single-threaded event model for editor mutations and from the fact that `BOARD_COMMIT` assumes it is operating on the current model, not arbitrarily reconciling parallel edits. ŅłĆciteŅłéturn25view0Ņłéturn26view0Ņłéturn17view2Ņłü

If accept-time replay fails, the live board must not end up half-updated. The safest way to guarantee that is: open one live `BOARD_COMMIT`, stage all replayed operations, run mandatory pre-push validation, and only `Push()` if staging and validation succeed. If anything fails before the push, drop the live transaction and preserve the preview session so the user can inspect or retry. That mirrors the stable public commit APIs and aligns with how `BOARD_COMMIT` separates staging from push and how `Revert()` restores state when a staged commit is abandoned. ŅłĆciteŅłéturn17view2Ņłéturn17view0Ņłéturn28view0Ņłéturn30view1Ņłü

Reject is simpler: discard the session preview state and the journal without touching the live board. Undo after accept is handled by KiCad natively because the whole accept is one live commit. Replay is just journal re-execution. In other words, **Reject** is a session-level discard, **Rollback** is a shadow-session time-travel to a checkpoint, **Undo** is a native live-board reversal after acceptance, and **Replay** is deterministic journal execution for debugging, reproducibility, or agent refinement. Keeping those four concepts separate will make the system much easier to reason about and much more resilient over time. ŅłĆciteŅłéturn17view0Ņłéturn16view6Ņłéturn28view0Ņłéturn30view1Ņłü

## Recommended architecture and MVP

The long-term stable architecture should be:

**Agent runtime layer** on top, potentially Python-first now but not Python-bound forever.  
**KiSurf session API** in the middle as the durable product boundary.  
**Typed atomic operation system** underneath, owning validation and journaling.  
**Shadow-board session core** beneath that for semantic truth.  
**Native KIGFX preview layer** beside it for rendering.  
**Final live-board promotion** through one `BOARD_COMMIT` at accept time.  
This layering is the best way to preserve future extensibility while ensuring that every mutation remains inspectable, replayable, rollbackable, and undo-integrated. It also mirrors the direction KiCad itself has taken in separating unstable raw bindings from a more stable process/API boundary. ŅłĆciteŅłéturn36search0Ņłéturn25view0Ņłéturn26view0Ņłéturn28view0Ņłü

For the **Python boundary**, the best pattern is: Python may do control flow, variables, loops, geometry calculations, and adaptive logic, but it should see only a **strict session API** such as `create_via`, `create_track`, `create_zone`, `update_item`, `query_board`, `render_preview`, `validate`, `checkpoint`, `rollback`, and `accept`. Python should never receive raw `BOARD*`, raw `BOARD_ITEM*`, or unrestricted access to KiCad internals. KiCadŌĆÖs own documentation now explicitly says the old PCB Python bindings are raw and unstable across major versions, while the IPC stack was designed to be stable and language-agnostic. That is the cleanest evidence that direct-language binding to internals is the wrong long-term product boundary, even if Python remains your preferred orchestration language. ŅłĆciteŅłéturn36search0Ņłéturn25view0Ņłéturn26view0Ņłü

For the **process model**, the most future-proof choice is an **out-of-process per-session Python worker** or a small pool of workers, with all board mutation executed synchronously through the KiSurf session owned by the editor process. KiCadŌĆÖs IPC API docs emphasize that KiCad is an event-driven GUI application with a single-threaded event-handling model for API requests, and the official plugin system now runs Python plugins in external interpreters and virtual environments. That is a strong precedent for keeping language runtime isolation while reserving actual model mutation and rendering ownership for the editor core. It also makes cancel, timeout, logging, and crash containment much easier than embedding a powerful general-purpose runtime directly into the UI process. ŅłĆciteŅłéturn25view0Ņłéturn26view0Ņłü

For the **preview layer**, introduce a dedicated `AI_PREVIEW_MANAGER` that owns AI preview `VIEW_ITEM`s, overlay annotations, step timeline state, and mappings from journal ids to preview ids. Use dedicated AI preview layers or groups within `KIGFX::VIEW`, rather than writing transient objects into the real board model. Use the shadow board to compute what should be shown; use the preview manager to render it; use the view/query APIs to support click-through inspection. When needed, connect DRC and validation overlays directly through `DRC_ENGINE::SetDebugOverlay`. ŅłĆciteŅłéturn12view0Ņłéturn12view1Ņłéturn11view4Ņłéturn39view5Ņłü

For the **MVP**, I would not start with arbitrary live/live concurrent editing. Start with a single modal AI session per board tab: create one shadow board from the current board, attach a typed journal, let the agent execute steps with preview, checkpoint, rollback, and validation, then accept via one `BOARD_COMMIT` if the base hash still matches. Include only a few atomic ops at firstŌĆövias, tracks, zones, move/update, object query, board-state query, validation, preview render, checkpoint, rollback. Defer cross-session rebasing, partial live-board merges, and speculative background mutation until the shadow-board and journal semantics are proven. That gets you to a system that is already architecturally correct, not a throwaway prototype that will need to be rebuilt later. ŅłĆciteŅłéturn10view0Ņłéturn17view2Ņłéturn28view0Ņłéturn30view1Ņłéturn25view0Ņłü

The single most important design rule is this: **do not define a separate API for every composite action**. CAD and EDA tasks compose too freely. The stable platform surface must be small, typed, semantic, and replayable. Everything above itŌĆöPython scripts, LLM plans, parameter tuning, step-by-step user-visible execution, and eventually other scripting runtimesŌĆöshould compile down to the same atomic operation journal against the same session core. That is what will keep the system extensible five years from now instead of trapping you in a growing pile of bespoke ŌĆ£AI toolsŌĆØ that are hard to compose, validate, or undo.

# KiSurf Next Action Implementation Strategy

## Executive architecture

The strongest implementation path is a **single in-editor Next Action agent** with one shared runtime and three work-state adapters—placement, routing, and auto-fill/refill—rather than three separate agents or an external IPC/plugin stack. That recommendation follows directly from how KiCad itself is structured today: the editor already exposes a unified board model, selection model, tool manager, connectivity database, property system, commit/undo machinery, and interactive router, while KiCad’s API-plugin model is explicitly oriented around **external processes communicating over IPC**. For KiSurf, that makes the external plugin path a useful reference point, but the wrong primary architecture. The production design should therefore stay inside the KiCad-based editor runtime and treat all deterministic subsystems as tools under one LLM-led orchestration loop. citeturn20search1turn20search4turn20search7turn11search11turn3search8turn17search0

Repository-specific connector retrieval for `wenming-ma/kisurf` was not available in this session, so the report maps your brief onto KiCad’s documented internals and APIs rather than line-by-line repository symbols. Within that constraint, the architectural fit is still clear: KiCad already supports transactional editing, item-level creation and update, connectivity/ratsnest rebuilds, hidden or deferred commit application, and native rendering/export surfaces. Those are exactly the primitives needed to turn your existing `AI_NEXT_ACTION_RUNTIME` foundation into a production Next Action engine. citeturn17search0turn6search13turn6search6turn3search8

The recommended runtime loop is this:

```text
semantic event
   -> work-state classifier
   -> observation packet snapshot
   -> LLM decision turn
   -> hidden attempt runtime
       -> atomic ops
       -> integrated tools
       -> bounded script plan
   -> render + validate
   -> LLM review turn
   -> preview lease publication
   -> accept-token guarded replay onto live board
```

Two design rules should remain non-negotiable. First, the LLM is the semantic decision-maker; routers, anchor generators, property schemas, overlap evaluators, and validators only return candidates, constraints, and diagnostics. Second, every candidate suggestion is born in a **hidden attempt context**, never as a live mutation. KiCad’s own board APIs and commit systems already distinguish between staged edits and published edits, which is exactly the seam KiSurf should exploit. citeturn17search0turn6search13turn15search13

## Layered tool surface

The three-layer tool model in your brief is the right shape and should be kept, but the layers need sharper contracts. The key refinement is that the **script layer should not be “Python access”** in the broad sense. It should be a bounded, journalable IR that may reuse your existing Python/session substrate for execution, but only after compiling down to approved atomic and integrated tools under Next Action runtime control. That preserves rollbackability, replay, preview-first behavior, and clear blame assignment in logs. KiCad’s own scripting and API surfaces are powerful, but the documentation explicitly warns that the scripting API is tightly coupled to internals and unstable across versions, which argues for a narrow KiSurf-owned tool ABI above raw editor APIs. citeturn1search6turn1search9turn1search15

A good production split looks like this:

| Layer | What it does | Recommended KiSurf contract |
|---|---|---|
| Atomic operations | Small typed reads and writes | Must be pure query or single-commit mutation against a hidden attempt context; must be journaled, diffable, rollbackable |
| Integrated tools | KiCad-native helpers and evaluators | May call multiple atomic ops and internal KiCad subsystems, but must return candidates/facts, never publish a user-visible suggestion directly |
| Bounded script plans | Compose new action sequences | Compiles to atomic/integrated invocations, with strict budgets on steps, diffs, time, and created objects |

The **atomic layer** should be boring and comprehensive. For PCB work, that means typed queries for board revision, active tool, selection, cursor, viewport, visible layers, enabled layers, nets, netclasses, design rules, stackup, connectivity, connected items, bounding boxes, pads, footprints, tracks, vias, and tables; plus typed mutations for create/update/remove items, set active layer, move footprint, rotate footprint, flip footprint, create track, create via, set row/column/table properties, and set item properties. KiCad’s board APIs already expose selection, items, nets, pads, connected items, bounding boxes, layers, commit transactions, and create/update/remove semantics. Footprints expose position, orientation, side changes, and layer-flip behavior, while tables expose explicit row/column/cell and border/separator mutators. citeturn17search0turn18search2turn18search0turn21search2turn21search5

The **integrated layer** should encode the places where KiCad knows more than a raw item setter does. For placement, that means tools such as `generate_placement_anchors`, `evaluate_courtyard_overlap`, `evaluate_keepout_conflict`, `score_ratsnest_delta`, and `summarize_connectivity_change`. For routing, it means `route_to_anchor`, `route_parallel_candidate`, `probe_router_start_points`, `generate_path_family`, and `summarize_route_quality`, all built on top of KiCad’s router internals rather than bespoke geometry search. For refill/property editing, it means `infer_patch_shape`, `fetch_property_schema`, `coerce_patch_values`, `validate_property_patch`, and `validate_table_patch`. That is consistent with KiCad’s own internals: the router has explicit start-routing and start-dragging entry points, the board exposes connectivity and rules, footprints expose courtyard and bounding-hull geometry, and properties are mediated through the property system rather than arbitrary string mutation. citeturn19search0turn19search2turn3search8turn8search6turn22search4turn22search6

The **script layer** should be deliberately constrained. Recommended limits for MVP are: no recursion, no unbounded loops, a maximum of 20 tool invocations per attempt, a hard wall-clock budget, a created-item budget, an object-touch budget, and a mandatory checkpoint at the start of every script. The language itself can be tiny: variables, `for each candidate`, `if diagnostic`, `checkpoint`, `rollback`, `choose best`, and `render`. The runtime should reject any compiled plan that tries to mutate live state, bypass validation, or open side channels outside the attempt context. That gives the model compositional freedom without letting it become an uncontrolled editor script runner. The need for that boundary is reinforced by the instability of raw scripting interfaces across KiCad versions. citeturn1search6turn1search9turn1search15

Journaling should be first-class. Every episode should record the semantic trigger, work state, observation packet hash, LLM prompt hash, LLM tool-call sequence, attempt context ID, checkpoints, rollbacks, tool arguments, result summaries, modified KIIDs, validation results, render artifact IDs, review decision, preview lease, accept/reject/expire outcome, and replay status. KiCad’s own commit and replay-adjacent infrastructure already treats grouped changes as a single undoable operation, and its diff/merge/property machinery shows that property-level change capture is feasible. KiSurf should extend that idea into a **Next Action journal**, not invent a second unrelated debug log. citeturn17search0turn15search13turn22search3turn22search8

## Observation packets and hidden attempts

The MVP observation packet should be **smaller than the model wants**, but richer than a plain selection dump. The minimum useful packet for all three work states is a structured object with: episode metadata, board revision, active tool/mode, cursor and viewport summary, selection summary, visible/enabled layer summary, concise net/netclass summary for touched items, relevant design-rule summary, connectivity/ratsnest summary, recent action summary, and one or two rendered images. KiCad’s board APIs already support retrieving the selection, visible and enabled layers, nets, connected items, bounding boxes, graphics defaults, stackup, and board render/export outputs, so this packet can be built without ad hoc scraping. citeturn17search0turn3search8turn3search13

For **placement**, the work-state overlay should add candidate target footprint IDs, pad summaries, local courtyard and keepout geometry summaries, nearby obstacles, local ratsnest deltas, side/orientation state, and anchor hypotheses. KiCad exposes footprint position/orientation/layer-flip operations, courtyard caches and bounding hulls, board-edge outlines, and connectivity/ratsnest APIs, so the LLM can be shown both the geometric and electrical context before it chooses a move. The practical reason to include both is that KiCad’s ordinary move operation can highlight courtyard conflicts without doing full DRC, which means a placement suggestion reviewed only visually will be under-informed unless KiSurf adds structured rule facts alongside the preview. citeturn18search2turn8search6turn8search8turn13search2turn13search11turn3search8

For **routing**, the overlay should add source/target anchors, incomplete airwires, current net/netclass constraints, active copper layer, nearby obstacles, local route corridor summaries, existing trace segments suitable for parallel routes, diff-pair context when present, and a compact summary of recent routing outcomes on the same net. This is important because KiCad’s interactive router is modeful and already supports push-and-shove, walkaround, collision highlighting, start-routing, and start-dragging behaviors. The model needs to decide **which router-assisted intent to try**, not paint every segment from scratch. citeturn0search1turn0search11turn19search0turn19search2turn13search5turn13search8

For **auto-fill/refill**, the overlay should add the active surface type, row/column/selection shape, per-field schema, available choices/enums, units, editability flags, and any validator diagnostics or coercion notes. KiCad’s property system already supports availability checks and properties with explicit choices, and PCB tables already expose cell collections plus row/column/table geometry. That means KiSurf can present the LLM with a compact typed editing surface instead of raw UI text. citeturn22search4turn22search6turn21search2turn18search0

A concise MVP packet schema can therefore be standardized like this:

| Packet block | Include in MVP | Why it matters |
|---|---|---|
| `episode` | Yes | Trigger source, work state, timestamps, revision IDs |
| `editor_state` | Yes | Active tool, selection mode, cursor, viewport |
| `object_summary` | Yes | Touched items, nets, layers, bboxes, local neighbors |
| `rules` | Yes | Netclass and applicable design-rule facts |
| `connectivity` | Yes | Airwires, connected items, local ratsnest health |
| `surface_schema` | Refill only | Table/property editability, types, choices, units |
| `attempt_artifacts` | After first attempt | Hidden render IDs, validation facts, created-item deltas |
| `history` | Trimmed | Last accepted/rejected suggestion and recent user edits |

Staleness handling should be strict and mechanical. Every packet should carry a **dependency fingerprint** composed of board revision, selection hash, active tool/mode hash, viewport/cursor region hash, and touched-object revision counters. A suggestion becomes unpublishable the moment any hard dependency changes, and unaccept-able the moment the accept token no longer matches the latest dependency fingerprint. This is stronger than “best effort freshness,” and it is necessary because KiCad’s board model is mutable, its selection and active tool are modal, and routing/placement state can materially change with a single user gesture. citeturn11search9turn11search18turn17search0

For hidden execution, the right design is a **new internal attempt runtime built on your existing session/checkpoint substrate**, not a free-floating reuse of the Chat Agent session and not a new external process. The implementation should feel like a dedicated `ATTEMPT_CONTEXT` with its own board handle, object map, checkpoint stack, render sink, and validation cache. Under the hood it can reuse the same rollback/journal machinery you already have, but it should have tighter invariants: one document, one episode, one work state, no direct publish path. That design matches both KiCad’s `BOARD_COMMIT`/revert pattern and the API-server transaction model where changes can be opened, grouped, pushed, or dropped. citeturn15search13turn6search13turn17search0

Rendering must also stay off-canvas until review succeeds. KiCad already supports rendered and plotted outputs from the board surface, and its GAL/view system is explicitly separate from item data. For KiSurf, the hidden attempt should render to an offscreen target or export artifact, feed that image back to the review turn, and only then allocate a native editor preview object tied to a preview lease. The user should never see attempt flashes caused by failed routes or bad placements. citeturn17search0turn12search2turn12search3turn12search4

Attempt budgets should differ by work state. A practical MVP is **up to three hidden attempts** for placement, **up to five topology attempts** for routing, and **up to two patch attempts** for refill. The LLM should get exactly one decision turn, the runtime should spend the allowed attempt budget, then the LLM should get one review turn with structured diagnostics plus the best render. That keeps latency bounded and makes silent failure a normal outcome instead of a reasoning spiral. The underlying reason to favor small budgets is that KiCad’s router and rule engines are already sophisticated; the point of the model is choosing promising intents, not brute-forcing the entire search space. citeturn19search0turn19search2turn13search4turn13search7

## MVP vertical slices

For **placement**, the first useful vertical slice should be “single-footprint semantic repositioning.” Start from one selected footprint, unresolved airwires, or a just-imported/just-moved footprint. The atomic tools needed are straightforward: query footprint state, query nearby items, query pads and nets, checkpoint, set position, set orientation, set side/layer-flip, render, validate, rollback. The integrated tools should do the real lifting: anchor generation from nearby pads and board edges, courtyard overlap checks, keepout/rule-area checks, local ratsnest delta scoring, and “would drag preserve pad-origin connections?” analysis. KiCad exposes exactly the geometry and item operations needed for this: footprints have position/orientation/layer-flip setters and cached courtyards/bounding hulls; the board exposes board outlines, pads, nets, and connectivity; and KiCad’s placement UX itself highlights courtyard conflicts even when the basic move command does not run full DRC. citeturn18search2turn8search6turn8search8turn13search2turn13search11turn3search8

Before previewing a placement, the review turn should receive a compact fact bundle: changed footprint reference, applied x/y/orientation/side, nearest-anchor relation, local courtyard collision count, keepout/rule-area collisions, delta in airwire count or total ratsnest length, and a confidence tag explaining whether the result is “clean,” “electrically improved but geometrically risky,” or “visually plausible only.” That is the minimum set that makes the model behave like a semantic reviewer instead of a blind image ranker. Courtyard validity and courtyard-overlap rules are public, documented KiCad concepts, so surfacing them explicitly will align KiSurf output with the editor’s own layout semantics. citeturn8search5turn8search9turn8search12

For **routing**, the first useful vertical slice should be “one-net route completion from anchor to anchor.” The model should not operate at pixel-by-pixel or per-segment raster level. It should choose between a small menu of intents: `route_to_pad`, `route_to_existing_trace`, `route_parallel`, `escape_then_complete`, `switch_layer_with_via`, `drag_existing_trace`, or `abandon`. Under the hood, the integrated tools should call into KiCad’s native router facilities—specifically the path where `PNS::ROUTER` supports `StartRouting`, `StartDragging`, world synchronization, and commit of routed items, while `ROUTER_TOOL` sits above it in the editor tool stack. That gives KiSurf access to the same routing intelligence a user gets from KiCad’s interactive router, including its shove/walkaround/collision-highlight modes, instead of reducing the LLM to low-level geometry scripting. citeturn19search0turn19search2turn2search2turn2search5turn0search1turn0search11

Routing validation should go well beyond pass/fail DRC. The review facts should include route completion status, remaining airwire distance, via count, total length, layer changes, minimum clearance margin encountered, whether router shoving was used, whether the candidate preserved diff-pair related intent when relevant, and a categorized violation summary if DRC found issues. KiCad’s DRC engine and tests already support structured violation handling, and the board design settings carry netclass-based track and via constraints. That means KiSurf can hand the model a rich scorecard instead of a binary red/green bit. citeturn13search4turn13search7turn3search13turn10search0

For **auto-fill/refill**, the first vertical slice should be limited on purpose: support PCB tables and high-value property editors before broader semantic panels. Tables are attractive because KiCad already exposes them as structured objects with cells, rows, columns, widths, heights, borders, and separators. Property surfaces are attractive because the property system already models availability and finite choices. In practical terms, the LLM should be allowed to emit only four patch forms for MVP: `set_cell`, `fill_row`, `fill_column`, and `fill_range`, plus a parallel `set_property` patch family for property editors. Schema providers should return field type, units, choices, nullability, read-only/editable state, and range limits. Validators should return diagnostics or normalized values, but never “decide for” the model whether to fill. citeturn21search2turn21search5turn18search0turn22search4turn22search6

The best long-term model for refill is to treat every editable surface as a **typed patch surface**. A table cell patch, a property-grid patch, and a semantic panel patch should all compile into the same internal representation: `target surface + selected region + field schema + patch values + validator feedback`. That lets one Next Action agent reason across all refill contexts with one tool vocabulary, instead of building three ad hoc refill stacks. KiCad’s widespread use of the property manager across board items, dialogs, and diff/merge machinery suggests that this normalization step is realistic and sustainable. citeturn22search0turn22search3turn22search5turn22search6turn22search12

## Scheduling, previews, and acceptance

The scheduler should never fire an LLM call directly from raw UI churn. In particular, **mouse-move events, drag ticks, live router hover updates, pan/zoom updates, and key repeats** should be consumed only by local debounce logic and work-state state machines. What should open a Next Action episode is a semantic event: selection became stable, a footprint move/drag ended, routing stopped with unresolved connectivity, a table edit committed, a property surface stabilized around a coherent selection, or a meaningful local board change occurred. KiCad’s editor is tool-modal and returns to the selection tool when a tool is canceled, which is a strong sign that Next Action should key off **stable editor states**, not raw event streams. citeturn11search9turn11search18turn13search8

The idle/stability thresholds should differ by work state. A good starting policy is 250–500 ms after a stable event for placement, 400–800 ms after routing pause/abort/unfinished-airwire stabilization for routing, and 150–300 ms after table/property edit commit for refill. Those windows are short enough to feel assistant-like, but long enough to avoid firing during ordinary hand motion. More importantly, they align with the fact that routing and placement usually have more expensive hidden attempts than refill does. The scheduler should also enforce a per-document cool-down so the agent does not immediately re-open an episode after the user rejects or ignores a suggestion on the same dependency set. citeturn0search1turn17search0turn19search0

Silence is a feature, not a failure mode. The agent should stay silent when the selection is too ambiguous, when the likely best action would require a disruptive multi-object edit beyond the current budget, when the hidden attempts produce only rule-violating or visually poor results, when the context changed during deliberation, or when the user is in a clearly active micro-editing streak. That policy is especially important in routing, where an unfinished net can tempt the system to over-suggest even when current topology is too constrained for a clean candidate. KiCad’s router and DRC subsystems already distinguish between collisions, shoving, and full rule checking; KiSurf should exploit those facts to decide not just what to suggest but also when **not** to suggest. citeturn0search1turn13search5turn13search7

The preview model should remain aggressively simple. There should be **one active Next Action preview per document**, owned by one preview lease, with a visible provenance tag pointing back to the episode and tool trace. The preview lease should expire on any conflicting edit, on selection/tool change affecting dependencies, on timeout, or when superseded by a newer higher-confidence suggestion. Acceptance should never “promote” the hidden attempt directly. Instead it should **replay the journaled tool trace** onto the live board only if the accept token still matches the dependency fingerprint. If it does not match, acceptance must hard-fail and optionally trigger regeneration. That is the cleanest way to prove stale suggestions cannot be applied. citeturn17search0turn15search13turn6search11

Ownership between the Chat Agent and Next Action should be document-scoped and explicit. The safest rule is that only one runtime may own a write-capable hidden attempt context per document at a time, while the other may observe but not mutate. A lightweight lock manager keyed by document ID and runtime purpose will avoid the worst class of interleaving bugs. This is particularly important because KiCad already has both editor-native commits and API/plugin paths that can affect board state; KiSurf should centralize ownership rather than let multiple AI pathways race. citeturn20search1turn20search4turn17search0

## Testing, evaluation, and major risks

The test plan should be built around **replayability first**, because replay is what will let you prove safety and improve quality at the same time. The canonical replay bundle should contain: semantic trigger, observation packet, work-state label, tool-call trace, checkpoint/rollback events, render artifact IDs, validation facts, review decision, published preview metadata, accept/reject/ignore signals, expiration reason, and final live-board diff. KiCad’s own API E2E fixtures, synthetic-board DRC tests, and board-commit tests show that the ecosystem already supports testable transactional editing and in-memory board exercises; KiSurf should piggyback on that style rather than rely only on brittle GUI tests. citeturn10search1turn10search0turn6search11turn15search13

The highest-priority correctness tests are the safety ones. You need explicit tests proving that hidden attempts never mutate the live board, that dropped/rolled-back attempts restore state exactly, that accept tokens fail after dependency changes, that preview leases expire correctly, and that replayed accepted edits produce the same live-board diff that the hidden attempt advertised. KiCad’s `BOARD_COMMIT::Push`/`Revert` behavior and commit-group semantics make these tests both feasible and meaningful. citeturn15search13turn6search13turn17search0

Quality evaluation should be separate from correctness evaluation. For placement, the main metrics should be accepted-preview rate, airwire reduction, local courtyard conflict count, keepout/rule violations, and edit compactness. For routing, use route-completion rate, remaining airwire reduction, via count, route length, DRC severity mix, and time to first acceptable preview. For refill, use schema-valid patch rate, value-correctness rate, user-edit preservation rate, accept rate, and retry rate. The common success metric across all three work states should be **accepted useful suggestions per editor hour**, because that is what will tell you whether Next Action is helping rather than merely being technically competent. Those metrics map cleanly onto KiCad’s exposed rule, connectivity, and item state surfaces. citeturn3search8turn13search4turn17search0turn21search2

GUI smoke tests still matter, but they should sit on top of the replay corpus rather than replace it. The right stack is: deterministic hidden-attempt unit tests, replay-based CI tests on synthetic boards and saved fixtures, then a small number of end-to-end GUI tests that verify preview presentation, accept/reject wiring, and canvas correctness inside the actual KiSurf editor. KiCad’s own test infrastructure already distinguishes lower-level API/E2E verification from editor behaviors, so mirroring that layering will keep your CI stable and informative. citeturn10search1turn10search3

The largest risks are architectural, not model-related. The first is **API and internal-surface churn**: KiCad’s scripting/API docs explicitly say the scripting API is unstable because it is tightly coupled to internals, so KiSurf needs its own stable tool ABI above raw editor APIs. The second is hidden-attempt state cost: cloning or checkpointing complex boards and rendering them offscreen can become expensive if you allow too many attempts. The third is router nondeterminism and side effects if you expose too much of the router’s low-level surface. The fourth is property-surface fragmentation if tables, dialogs, and semantic panels are not normalized into one typed patch model. The fifth is cross-runtime contention with Chat Agent sessions. Those are all solvable, but each one deserves an explicit engineering owner before the MVP is expanded. citeturn1search6turn1search9turn1search15turn19search0turn20search1

The main unresolved design decision is **how to materialize hidden attempts**. There are three plausible implementations: full shadow-board clone, checkpoint/undo branch against the live board object graph, or a lightweight change-set overlay. Based on KiCad’s existing commit and revert semantics, the best near-term choice is a checkpoint/undo-style hidden context if it can be proven non-leaky; otherwise, use a full shadow-board clone for correctness in MVP and optimize later. The second unresolved decision is whether offscreen preview rendering should use an internal GAL path or an export/render path first. The third is how much router surface to expose at MVP; the safest answer is “intent-level tools only.” Those choices should be settled before broadening beyond single-footprint placement, one-net routing, and table/property refill. citeturn15search13turn17search0turn19search2

In short, the implementation strategy is to keep one LLM-led Next Action runtime, expose a narrow KiSurf-owned three-layer tool ABI, build a strict hidden-attempt context with checkpoint/render/validate/review/publish gates, and ship three thin but real vertical slices: single-footprint semantic placement, anchor-level native-router-assisted routing, and typed patch refill for tables and property surfaces. That approach aligns with KiCad’s existing board, commit, router, property, and rendering internals while respecting your core principle that deterministic systems are tools, not decision-makers. citeturn17search0turn3search8turn19search0turn22search4turn20search1

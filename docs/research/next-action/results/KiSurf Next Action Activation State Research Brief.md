## Routing activation state

For routing, the evidence supports a two-layer model almost exactly in the form you proposed. The deterministic outer state should be `routing_active`, beginning when the router tool is entered and a successful interactive session starts, and ending only when the route is committed and finished, canceled, or superseded by tool activation/switch. KiCadÔÇÖs router does this explicitly: `prepareInteractive()` calls `StartRouting(...)`; the routing loop then keeps processing `Motion`, `UndoLastSegment`, `routerAttemptFinish`, `routerContinueFromEnd`, left-click/drag commit actions, posture flips, and finish/cancel/tool-activation events; when the loop exits it calls `CommitRouting()` and `finishInteractive()`, which stops routing, resets layer/cursor/autopan, and drops the undo block. KiCadÔÇÖs public docs mirror these boundaries: `End` fixes all tracks and exits the router; `Backspace` undoes the last fixed segment; switching layers inserts a via and routing continues on the new layer; pressing `V` again before committing the via cancels that via placement and returns to the original layer. ţłÇciteţłéturn24view0ţłéturn26view1ţłéturn25view0ţłéturn27view0ţłéturn28view0ţłéturn11search5ţłéturn16search0ţłü

Inside that outer state, KiSurf should use a second state such as `suggestion_ready`. That inner state should not track ÔÇťthe user is routingÔÇŁ; it should track ÔÇťthe current route snapshot is stable and meaningful enough to justify a hidden attempt.ÔÇŁ The source makes clear why this second layer matters: on every motion event KiCad updates the endpoint and calls `Move(...)`, so the tentative route is changing continuously at interactive frame-rate; if KiSurf spawned an attempt on every motion frame, it would thrash badly. A better boundary is to treat these deterministic moments as candidate triggers for an attempt: a segment or corner was just fixed with `FixRoute(...)` and the route remains unfinished; `routerAttemptFinish` was invoked but did not complete and the router warped the cursor to the best reachable point; `routerContinueFromEnd` changed the start point to the other ratsnest end; the routerÔÇÖs failure reason surfaced after start or after a blocked operation; or the cursor entered a small, stable neighborhood near a plausible target anchor while current layer, width, posture, and anchor hypothesis remained unchanged for a short local window. That last dwell window is useful, but it should be a secondary stability check layered on top of routing state, not the primary state definition. ţłÇciteţłéturn27view0ţłéturn28view0ţłéturn24view1ţłéturn24view0ţłéturn25view0ţłü

What the Agent should observe in `routing_active` is also well supported by the native model. The router already maintains the current mode, current layer, whether a via is being placed, the current start and end snap points, the start and end items, whether routing is in progress, and whether the user has flipped posture or switched route mode. KiCadÔÇÖs docs add the relevant user-level semantics: current net or coupled differential pair, router mode such as shove, walkaround, or highlight collisions, width/via sizing, and concrete finish/continue commands. Those are the right primary signals for KiSurf observation: current net context, route start anchor, committed stub, current tentative end, candidate target anchors, current layer, current mode, and ÔÇťblocked versus trivially completableÔÇŁ status. ţłÇciteţłéturn19view0ţłéturn28view0ţłéturn16search0ţłéturn11search0ţłü

The strongest routing recommendation is therefore this: publish previews mostly at commit boundaries, not continuously. The highest-value publish points are after a corner or segment commit when the route remains unfinished, after a failed or partial attempt-to-finish, and after a stable hover near a plausible target anchor that is not changing every frame. During raw cursor motion, the Agent should usually stay in observation-only mode. It should also suppress intervention while active shoving/dragging is occurring, when the tentative geometry is changing frame-by-frame, or when the user is already one trivial click away from completing the route manually. The source already encodes some of the reasons: motion drives `Move(...)` every frame; router mode can change during the interaction; layer switches and via placement mutate the routing state; and the tool treats finish, cancel, inline drag, and activation of another tool as boundaries that break the current flow. ţłÇciteţłéturn25view0ţłéturn25view1ţłéturn27view0ţłéturn28view0ţłéturn16search0ţłü

For hidden-attempt supersession, the right rule is aggressive invalidation. Any routing attempt or preview should be canceled or marked stale when the route endpoint changes materially, the current layer changes, via placement is entered or canceled, posture flips, width/via settings change, undo-last-segment fires, continue-from-end fires, finish succeeds, cancel fires, or another tool becomes active. That follows directly from KiCadÔÇÖs event architecture, where each of those events changes the legal route search space or the intended user trajectory. ## Placement activation state

For placement, the right outer state is likewise deterministic: `placement_active` should begin when a placement or move/drag command arms the tool, or when an object becomes attached to the cursor, and it should end on commit, cancel, or tool switch. KiCadÔÇÖs public docs and tool code both match that structure. The footprint placement tool is explicitly described as click-to-open-chooser, then click-to-confirm-location; free via placement is click-to-place in the editing canvas; and the generic drawing tools expose one guarded interactive session per command, turning autopan and cursor capture on only once the interaction is active and resetting cursor state on exit. AltiumÔÇÖs current PCB placement documentation says the common pattern is broadly the same regardless of object type: select the object to place, use the mouse to define its location and size if applicable, and terminate placement mode with right-click or `Esc`. ţłÇciteţłéturn33search0ţłéturn40search6ţłéturn31view0ţłéturn31view1ţłéturn40search3ţłü

That means your proposed layered model is a very good fit: `placement_active`, then `candidate_context_stable`, then `preview_publishable`. In `placement_active`, KiSurf should observe continuously but stay visually quiet. It should wait for `candidate_context_stable` before starting a hidden attempt. That inner stability state should mean the attached item identity is unchanged, the transform basis is stable enough to matter, the cursor is no longer sweeping quickly, nearby candidate anchors or conflicts are coherent, and the user appears to be evaluating rather than merely moving. This is especially important because placement is often more ballistic than routing: a component, via, or graphic can traverse large parts of the board rapidly, and publishing early would feel like the system is ÔÇťfighting the hand.ÔÇŁ AltiumÔÇÖs generic placement guidance and KiCadÔÇÖs armed-tool model both support the idea that the semantic state changes at arming and commit/cancel, while any timing logic belongs only inside the stability gate. ţłÇciteţłéturn40search3ţłéturn31view0ţłéturn31view1ţłü

The set of start events should be broader than just ÔÇťadd footprint.ÔÇŁ It should include selecting a placement tool, selecting a footprint from a library or picker, starting a via placement, starting a graphic/shape/zone/keepout placement, and starting a move or drag of an existing item. KiCadÔÇÖs docs explicitly cover free via placement, footprint placement, and dragging footprints while keeping tracks attached; KiCadÔÇÖs router settings also state that the router mode applies when dragging existing tracks, and footprint dragging in highlight-collision mode cannot be committed if it creates DRC violations unless the user explicitly forces the commit. This is important for KiSurf because ÔÇťmove or drag an existing itemÔÇŁ is not semantically the same as ÔÇťstatic selection existsÔÇŁ; it is a live transformation command with a definite commit boundary and a definite legal-state test. ţłÇciteţłéturn40search6ţłéturn33search0ţłéturn16search0ţłéturn40search2ţłü

The most defensible preview moments in placement are not arbitrary pauses. They are moments when the item is attached to the cursor and the local situation becomes interpretable: the item enters a neighborhood with a strong anchor candidate; the item approaches an obstacle or conflict and the tool is likely to reject a naive commit; the user has just placed a similar item and a continuation pattern exists; or the current local orientation/side choice is probably suboptimal but still legal. AltiumÔÇÖs object-placement commonality and KiCadÔÇÖs attached-item placement patterns both suggest that placement support should feel like ÔÇťyou are here, and this nearby landing zone looks materially better,ÔÇŁ not like constant mid-flight correction. ţłÇciteţłéturn40search3ţłéturn33search0ţłéturn40search15ţłü

For multi-place handling, the most robust recommendation is conditional continuity. If the native placement tool remains armed after a commit, then `placement_active` should continue and only the snapshot token should advance. If the native tool drops back to idle after commit, `placement_active` should end immediately and the scheduler should wait for the next deterministic arm event. This avoids guessing a multi-place workflow that the editor did not actually enter. The same rule should apply across vias, repeated graphics, and any footprint/library placement flow. That recommendation is an inference from how explicit placement-mode termination is documented in Altium and how KiCadÔÇÖs drawing tools wrap each interactive command in a bounded event loop. ţłÇciteţłéturn40search3ţłéturn31view0ţłéturn31view1ţłü

## Auto-fill and refill activation state

Auto-fill and refill is the one domain where simple tool-state metaphors are least sufficient, because the user is often editing in typed surfaces rather than driving a geometric cursor loop. Here the right outer state is `surface_active`: focus is inside an editable, project-semantic surface such as a property panel, typed dialog field, netclass table, rule table, BOM-like grid, or inspector. But `surface_active` is not enough by itself. Standards and UI platform docs show why: text editing emits composition events, `beforeinput`/`input`, selection changes, and change/validation events with different timing semantics. The W3C UI Events and Input Events specifications, plus MDNÔÇÖs API references, distinguish IME composition (`compositionstart` through `compositionend`), editable text mutations (`beforeinput` and `input`), selection changes, and final value changes (`change`) that do not necessarily occur on every keystroke. Windows Forms validation documentation likewise treats validation as its own event boundary that often occurs on focus change. ţłÇciteţłéturn34search1ţłéturn36search0ţłéturn36search1ţłéturn36search2ţłéturn36search14ţłéturn34search7ţłéturn34search3ţłü

That supports the layered model you proposed almost verbatim: `surface_active` Ôćĺ `edit_in_progress` Ôćĺ `commit_observed` Ôćĺ `selection_stable` Ôćĺ `preview_publishable`. `edit_in_progress` should be entered whenever the focused field or cell is actively being edited, and it should suppress suggestions. In practical terms, suppression should hold while the user is typing, while IME composition is active, while the user is selecting text inside the field, while a popup editor or dropdown remains open, and while validation is still running or may still reject the edit. AG GridÔÇÖs editing lifecycle is a useful analogue here: editing starts and stops on explicit triggers such as Enter, Esc, Tab, and popup-editor closure; editing stop and value-change are distinct events; and value changes can also arise from paste, fill handle, copy range down, undo, and redo. For Auto-fill specifically, the best trigger is usually not ÔÇťthe user paused mid-edit.ÔÇŁ It is ÔÇťthe user just committed something that plausibly establishes a local pattern.ÔÇŁ Spreadsheet tools make that distinction explicit. Fill-handle systems in spreadsheet-like grids likewise operate after a cell or range is selected and then extended, not simply when the caret is idle. For preview form, the best UI depends on scope. A single-cell pattern should use an inline ghost value or inline diff. A local refill across adjacent cells should use a highlighted cell-diff preview. A broader structured patchÔÇösuch as a row bundle, a property bundle, or a netclass/rule-table patchÔÇöshould render as a grouped patch preview with explicit accept/reject controls. Auto-fill staleness should also be stricter than ordinary inline text completion. A suggestion should go stale when the focused cell changes, the selected range changes, the editable surface changes, source values change, schema or validation rules change, the user begins a new typing/composition session, or the project context changes enough that the suggested patch was derived from the wrong nearby evidence. ## Recommended scheduler and instrumentation

The cleanest `AI_NEXT_ACTION_SCHEDULER` is a domain-and-phase model, not a timer-centric trigger tree. A practical unified shape is:

```text
domain = idle | routing | placement | surface
phase  = observing | suppressed | attemptable | hidden_attempt | preview_ready

common tokens:
- session_id
- snapshot_id
- generation_id

routing snapshot keys:
- route_mode, net_id, start_anchor, end_anchor, layer, via_state, posture, width/via settings

placement snapshot keys:
- tool_kind, item_id, transform, layer/side, drag_kind, local anchor set, legality status

surface snapshot keys:
- surface_id, field_or_range_id, schema_hash, edit_state, selection_hash, validation_state
```

This scheduler should be driven first by native state transitions and commit/cancel events, and only secondarily by local stability windows. For routing, native signals should drive entry, exit, route continuation, finish, undo, layer change, via placement, router mode change, and drag/inline-drag boundaries. For placement, native signals should drive tool arm, object attached-to-cursor, drag start, rotate/flip/side change, commit, cancel, and whether the tool remains armed after commit. For typed surfaces, native UI signals should drive focus-in/out, edit start/stop, popup-open/close, validation-start/done, change/commit, paste, and selection/range change. The hidden attempt should only start when the scheduler is in `attemptable`, and previews should publish only after deterministic legality checks and semantic review pass. That is directly aligned with how KiCadÔÇÖs router and drawing tools already expose stateful command loops and how table editors expose commit and validation events. ţłÇciteţłéturn25view0ţłéturn27view0ţłéturn28view0ţłéturn31view0ţłéturn31view1ţłéturn34search2ţłéturn34search7ţłü

The role of timing should be intentionally narrow. In routing, time should be used for noise reduction only: cursor-velocity filtering, local endpoint stability checks, and visual publication throttling. In placement, time should mainly help the system avoid speaking too early while the object is still flying across the canvas. In Auto-fill, time is somewhat more useful, but still mostly as a post-commit settling window after focus/range/validation changes. KiCadÔÇÖs own source reinforces this separation by using a 200 ms UI-update guard to reduce routing overhead rather than to define when routing is ÔÇťactive.ÔÇŁ VS CodeÔÇÖs inline suggestion configuration similarly treats delay as a presentation control, not a semantic interaction-state definition. ţłÇciteţłéturn25view0ţłéturn26view1ţłéturn37search3ţłü

The split between ÔÇťnative signalsÔÇŁ and ÔÇťnew instrumentationÔÇŁ is also fairly clear. Native KiCad/KiSurf signals should include tool entry/exit, push/pop tool operations, route in progress, route commit/cancel/finish, layer changes, via-placement phase, router mode, dragging mode, drawing-tool active state, board-commit success/failure, and any focused-widget edit/commit/validation signals your UI toolkit already exposes. New instrumentation is most needed for things the editor likely does not expose semantically today: cursor velocity and dwell buckets, anchor-proximity scoring, preview supersession tokens, hidden-attempt cancellation tokens, typed-surface taxonomy, IME composition hooks if the current widget layer does not already surface them, popup-editor open state, schema hashes, and local pattern-detection summaries for refill candidates. That keeps deterministic observation infrastructure separate from the semantic Agent itself, which matches your constraint that the detector is scheduler infrastructure, not the decision-making Agent. ţłÇciteţłéturn19view0ţłéturn25view0ţłéturn31view0ţłéturn34search1ţłéturn36search1ţłéturn34search2ţłü

## Final answers

**Routing should be primarily driven by interactive routing tool state, not timing thresholds.** The KiCad source is unusually explicit here: router activation, route-in-progress, move, commit, undo, finish, cancel, and tool-activation boundaries already exist as first-class events and methods. **Placement should be primarily driven by tool-armed state, attached-item state, and commit/cancel boundaries.** That applies to footprint placement, via placement, shape/graphic/zone placement, and move/drag of existing items. **Auto-fill needs an edit-transaction model, not a raw pause model.** The correct event stack is focus into a project-semantic editable surface, edit start, IME/text-edit suppression, edit commit or validation acceptance, selection/range stabilization, then preview publication. **Observation, update, and cancel/supersede should be separated.** Observation should run continuously inside `routing_active`, `placement_active`, or `surface_active`. **Timing still matters, but as support infrastructure.** Use it for local stability checks, publication throttling, and rate limiting; do not use it as the main semantic definition of routing or placement activity. **The recommended scheduler is a domain-and-phase state machine.** Use `domain = routing | placement | surface` and `phase = observing | suppressed | attemptable | hidden_attempt | preview_ready`, keyed by deterministic session and snapshot tokens.

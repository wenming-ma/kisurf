# KiSurf Runtime Architecture for an LLM-Mediated Next Action Agent

KiSurf already has the right substrate for AI-native editing: a session service, typed atomic operation journal, semantic shadow board, native preview bridge, validation bridge, rollback/checkpoints, and a one-transaction accept path into the live board. The architectural question for Next Action is therefore not “how do we bolt on a heuristic suggester,” but “how do we make every suggestion emerge from an LLM-controlled observe–act–render–review loop on top of that substrate.” In the current branch, the durable boundary is explicitly defined as Python SDK or agent cells feeding the KiSurf session service, typed atomic operation journal, shadow board executor, native preview manager, and final accept replay; preview must not mutate the live board, and accept is core-owned rather than Python-owned. fileciteturn2file0L7-L29 fileciteturn2file0L39-L45

My recommendation is a **single unified Next Action runtime with one reasoning locus**. In practice, that means the LLM remains the decision-maker for whether to suggest, what kind of action to attempt, which tools to call, whether an internal attempt is good enough, whether to retry or roll back, and whether to publish a final preview. Deterministic routers, validators, renderers, DRC engines, candidate generators, and geometry solvers still do most of the mathematical work, but they do so as tools under agent control rather than as parallel decision systems that can publish directly. That framing matches the current KiSurf product split between a user-directed Chat Agent and an ambient Next Action Agent that share underlying infrastructure, and it also matches the broader agent literature’s definition of a single-agent system: one LLM-controlled sequential loop can still use rich tools, self-reflection, and multiple iterations without becoming a separate multi-agent architecture. fileciteturn5file0L54-L92 citeturn14view7turn1search0turn1search1turn1search2

There is one meaningful internal tension worth calling out. An earlier KiSurf research note on Next Action recommended a hybrid stack in which deterministic providers and algorithmic candidate generators do the geometry-heavy work while models mainly rank and decide whether to surface suggestions. That computational insight is sound. The change I recommend is **not** to move DRC or routing math into the LLM; it is to move **publication authority and retry orchestration** into the LLM loop. In other words: keep the deterministic solvers, but prevent them from becoming autonomous publishers that bypass the agent. fileciteturn4file0L5-L19

## Recommended architecture

The best long-term shape is a **constrained single-agent loop** that runs as a special profile on the same session runtime already used by Chat Agent sessions. The loop should look like this:

```text
Observe current workspace snapshot
→ LLM decides: suggest now, wait, or stay silent
→ LLM chooses tool(s) and opens/continues a constrained shadow session
→ Runtime executes in shadow state only
→ Runtime renders, validates, and summarizes results
→ LLM reviews that result
→ LLM decides: publish / adjust / rollback / retry / abandon
→ If published: user sees one native preview
→ If accepted: runtime replays journal into live board in one commit
```

This is directly compatible with the current KiSurf runtime, which already exposes session opening and closing, step boundaries, checkpoints, rollback, observation, board queries, preview rendering, validation, and accept or reject; it also already treats composite helpers as SDK-level constructs that lower into atomic operations rather than proliferating model-facing one-off commands. That is exactly the right starting point for a Next Action runtime, because it preserves a small stable control surface while allowing many future composite behaviors to emerge from tool composition instead of hardcoded flows. fileciteturn2file0L57-L83 fileciteturn2file0L96-L145

The cleanest responsibility split is this:

| Layer | What it should own | What it should not own |
|---|---|---|
| **LLM Agent** | Opportunity judgment, work-state interpretation, tool choice, stop/continue decision, retry policy, preview publication, abstention | Raw pointer mutation, renderer internals, DRC internals, autonomous commit path |
| **Tools** | Candidate generation, geometry solving, path planning, validation, rendering, querying, scoring, diff extraction | Final publish decision, silent commit, cross-tool policy orchestration |
| **Runtime** | Session lifecycle, isolation, versioning, checkpoints, rollback, preview visibility, audit trail, accept replay, cancellation, resource budgets | Semantic decision of “this is the right next action” |

That split is strongly supported by the current generation of agent-engineering guidance. OpenAI’s tool-calling guidance treats tools as functionality the model can invoke, with strict JSON-schema contracts to keep execution typed and bounded; its reasoning cookbook explicitly notes that reasoning models may need serial function calls in a loop whose length is not known in advance. Anthropic’s guidance similarly emphasizes simple composable agent patterns, clear tool namespaces, meaningful tool return context, and token-efficient responses. citeturn14view1turn14view0turn14view3turn14view4turn14view5

For KiSurf, this implies one important rule: **non-decisional native gates are fine; decisional native publishers are not**. It is reasonable for the runtime to debounce events, suppress obviously unchanged snapshots, manage caches, enforce time budgets, and cancel stale attempts before the LLM sees them. Those are infrastructure controls. But once the system is making a semantic decision such as “there is a useful next action here,” “route candidate B is better than A,” or “show this preview now,” that decision should belong to the LLM-controlled loop. That preserves the user’s requested “LLM as the brain” model while still allowing the runtime to remain fast and safe. citeturn14view4turn14view7turn14view8

## Runtime loop and hidden-attempt mechanics

The runtime should treat **observation** as a first-class step, not as a passive serialization dump. In the current KiSurf spec, the observation/query vocabulary already includes board summary, item queries, selection, nets, layers, design rules, viewport, activity timeline, preview rendering, and step observation. The implementation also already returns recent editor activity and a session-journal summary through `query_activity_timeline`, and it rejects stale worker responses whose session id, board id, base hash, or epoch no longer match the request. That gives Next Action exactly the ingredients needed for an observation-driven loop rather than a text-only prompt. fileciteturn2file0L111-L124 fileciteturn3file0L63-L90 fileciteturn3file0L202-L235

For **internal self-review**, the right implementation is to run every attempt inside a scratch or shadow session with explicit checkpoints. KiSurf already has the crucial primitives: semantic shadow state, checkpoints, rollback, preview rendering, automatic validation/preview feedback after mutations, and journaled operation results. The runtime should therefore treat a Next Action suggestion as a bounded internal episode: open or reuse a constrained session, create a checkpoint, apply one or more attempt tools, request render and validation feedback, let the LLM inspect the result, and then either publish, roll back and retry, or abandon. Because the journal is already the canonical replay and audit artifact, this design keeps every internal attempt recoverable and inspectable without ever touching the live board. fileciteturn2file0L163-L184 fileciteturn3file0L177-L201 fileciteturn3file0L184-L193

The **difference between hidden attempts and published preview** should be explicit in the runtime model. I recommend reusing the same preview infrastructure for both, but with different visibility scopes and stronger provenance tags. KiSurf’s preview architecture already points in this direction: the preview manager tracks session-level groups, optional step-level groups, itemized preview objects, overlays, and provenance metadata; the preview service already renders the current shadow board for the editor and clears preview state on rollback, reject, cancel, close, or accept; rollback already clears the current preview and restores checkpoint preview state by re-rendering against the rolled-back shadow board. That is nearly everything needed for a two-tier preview system. The missing distinction is policy: **hidden attempt previews are for LLM inspection only; published previews are the only previews the user should normally see**. fileciteturn2file0L212-L239 fileciteturn3file0L137-L150

That distinction also fits native EDA interaction patterns. KiCad’s interactive router already distinguishes temporary route objects from committed objects: while routing, unfixed temporary segments are shown as brighter temporary geometry, and only fixed segments are saved; leaving the router or cancelling does not save the unfixed geometry. For KiSurf, the analogy is useful but should be pushed one step further: the engineer should usually see **only the final reviewed suggestion**, not the agent’s exploratory flailing. Intermediate attempts are valuable for the agent’s own review loop, but surfacing them by default would create flicker, attention theft, and false confidence. I would reserve visible “show alternatives” or “show reasoning process” modes for explicit user request, debugging, or cases where comparing two near-equivalent candidates is itself the useful product interaction. citeturn5view0turn5view1

The publish-state machine should be simple and strict. A suggestion starts as **observed**, becomes **attempting**, then either reaches **published** or is **abandoned**. A published suggestion can end as **accepted**, **rejected**, **expired**, or **superseded**. “Expired” means the context version it was bound to is no longer valid. “Superseded” means a newer suggestion for the same locus replaced it. “Rejected” is an explicit user refusal. “Abandoned” is an internal silent drop because the LLM judged the attempt low-quality or stale before publication. This kind of process-level traceability is not just nice to have; provenance research increasingly treats trace and execution lineage as the accountability layer for trustworthy agent systems, covering debugging, audit, recovery, and guardrails. fileciteturn5file0L88-L92 fileciteturn3file0L23-L41 citeturn14view6

## How the loop should behave across placement, routing, and auto-filling

A unified runtime is the right design, but the three dominant work states do require different observation packs, tool shapes, and self-review criteria. The key is to keep those differences at the level of **state-specific tool composition**, not at the level of separate agents. KiSurf’s own README already frames Placement, Routing, and Auto-filling or Refilling as the taxonomy of the Next Action Agent rather than three disconnected systems, and the current runtime already seeds typed shadow handles for tracks, vias, zones, shapes, footprints, and pads, including selected-item metadata needed for context-aware suggestions. fileciteturn5file0L73-L92 fileciteturn3file0L165-L176

For **placement**, the LLM should reason primarily over **relationships**, not raw coordinates. The observation pack should emphasize the selected or recently touched components, connected nets, local board region, relevant keepouts and mechanical boundaries, nearby components, routing pressure indicators, grid, symmetry opportunities, and whether the engineer is continuing an existing pattern. KiCad already exposes the editor as an integrated placement-and-routing environment with DRC, scriptable rules, cross-probing between schematic and PCB, grid control, and selection-centered navigation; KiCad 9 also introduces component classes and multichannel layouts, which are especially relevant to repeated placement patterns and room-like grouping. In a KiSurf Next Action runtime, the LLM should ask tools for candidate placements or relative anchors, then trial-place those candidates in shadow state, render them, check overlap, keepout, rule, and connectivity consequences, and retry if needed. The final preview should usually be a ghosted placement with orientation, anchor cues, local nets or ratsnest impact, and any warnings that matter to acceptance. citeturn5view1turn5view2 fileciteturn2file0L125-L145

For **routing**, the most important design choice is the abstraction level at which the LLM operates. My recommendation is a layered model in which the LLM usually speaks at the level of **intent, anchors, and candidate selection**, while the deterministic router remains the geometry constructor.

| Routing abstraction | Best use in KiSurf | Strengths | Main failure mode | Recommended role |
|---|---|---|---|---|
| **Coordinate-level** | Tiny corrective edits, endpoint nudges, geometry cleanup | Maximum precision | Brittle, high-token, poor generalization, easy to go stale | Fallback only |
| **Anchor/reference-level** | Connect this pad/via to that anchor or corridor | Compact, explainable, stable under viewport changes | Still under-specifies style in crowded regions | Core interface |
| **Candidate-path selection** | Choose among router-generated route options | Keeps LLM out of low-level geometry; easy to review | Depends on good candidate diversity | Core interface |
| **Intent-driven** | “Continue this route on top layer, minimize vias, preserve future bus corridor” | Best semantic control, good explainability | Needs strong downstream compiler/router | Core interface |
| **Router-as-tool** | Actual path construction and repair, DRC-safe geometry, diff pairs, push-and-shove | Deterministic legality and speed | Can become an autonomous decider if allowed to publish directly | Essential tool, never final publisher |

This recommendation is grounded in how modern PCB editors already work. KiCad’s interactive router supports guided routing of single tracks and differential pairs, push-and-shove and walkaround behaviors, respect for design-rule width and clearance, automatic tune operations, and “Attempt Finish” actions that route toward the nearest ratsnest anchor. That means the editor already contains a strong deterministic routing substrate; KiSurf should not replace it with token-by-token coordinate generation. Instead, the LLM should interpret the engineer’s local intent, choose routing tools and constraints, inspect router output, and decide whether the result is worth publishing. For buses, differential pairs, and timing-sensitive nets, the tool surface should become *more semantic*, not less: route group, lane order, spacing regime, via tolerance, skew target, open corridor, and no-go regions are much more durable interfaces than explicit per-segment coordinates. citeturn5view0turn5view1turn5view2

The routing self-review loop should therefore inspect more than “DRC passed.” It should also inspect clearance envelopes, via count, layer changes, remaining ratsnest, local congestion, whether the route follows the user’s apparent direction of travel, whether it harms likely future routability, and whether it matches the current style of nearby routing. DRC and connectivity tools should return structured facts; router tools should return not only route geometry but also summaries of why that geometry was chosen and what tradeoffs it made. The LLM then decides whether to keep, revise, or discard the attempt. This is exactly the kind of “tool returns meaningful context back to the agent” pattern recommended in current tool-design guidance. citeturn14view5turn14view6

For **auto-filling or refilling**, the same runtime applies but the loop can be much lighter. KiCad 9 now supports tables in the schematic and PCB editors, and KiSurf’s Next Action scope explicitly includes table completion and contextual filling. In this work state, the observation pack should focus on the active structured surface: field schema, allowed values or ranges, neighboring rows or panels, analogous values elsewhere in the project, recent edit history, and the smallest relevant project facts. The tools here are mainly query, schema-validation, consistency-check, and candidate-generation tools rather than geometry solvers. The LLM should still remain in the loop, but usually one short internal review is enough: generate a candidate value or patch, validate against schema and cross-field consistency, and publish a lightweight preview such as an inline ghost value or a diff badge. As soon as the user types into that field, changes selection, or changes the governing context, the suggestion should expire. fileciteturn5file0L79-L92 citeturn5view2

## Staleness, latency, and coexistence with Chat Agent

Staleness should be treated as a **first-class runtime condition**, not as a UI afterthought. KiSurf already binds session correctness to base hash, epoch, and selection revision, and it already rejects stale or cross-session worker output before it can touch shadow state. The spec also makes selection drift explicit: `QuerySelection()` includes selection revision metadata and `kisurf_accept_session` may refuse with `selection_conflict` if the live selection changed after session open. For Next Action, that same mindset should apply to every published suggestion. A suggestion key should include at least: board base hash, selection revision, active tool or mode, anchor handles, relevant viewport fingerprint, and any active form or table revision. If any of those move outside policy bounds, the preview must expire or be superseded before the user can accept it. fileciteturn3file0L23-L46 fileciteturn3file0L63-L90 fileciteturn5file0L88-L92

This strictness is justified technically and ergonomically. Recent research on Next Edit Suggestions in AI IDEs found that NES systems rely on broader context and are sensitive to transactional edits and human-IDE interactions, which increases the importance of context freshness and invalidation. EDA suggestions are at least as context-sensitive as NES in code editors, and often more so because geometry, connectivity, constraints, and visible spatial placement all matter simultaneously. A stale ghost route or stale placement preview is not a harmless typo; it is a false affordance on the canvas. citeturn13academia2

The simplest cancellation rule is: **cancel early, publish late**. During internal attempts, the runtime should perform a cheap freshness check before each mutating tool application and again before publication. If the context changed meaningfully, the attempt should be cancelled and either silently dropped or restarted from a fresh observation. Published previews should be sticky enough to avoid flicker, but not so sticky that they survive obvious context changes. As a product policy, I would start with very small internal retry budgets for background suggestions and only allow longer multi-attempt loops when the work state is unusually stable or the engineer has paused in a way that signals interest. That is a design recommendation rather than a claim from the sources, but it follows directly from the repo’s stated goal of low-friction ambient guidance and from general agent guidance favoring simple bounded loops over sprawling autonomous background behavior. fileciteturn5file0L73-L92 citeturn14view4turn14view8

The relationship with **Chat Agent** should be implemented as shared substrate plus policy separation. The README already states that Chat Agent and Next Action Agent share editor context, preview, validation, and operation infrastructure but serve different product roles, and the runtime spec currently allows only one active AI execution session per board tab in the MVP. The clean near-term policy is therefore a **lease model**: Chat Agent gets exclusive access to any mutating or published preview session while it is actively running; Next Action downgrades to read-only observation or pauses publishing until the lease is released. Both agents should share the same capability vocabulary, session runtime, preview manager, validation, rollback, and accept path, but with different budgets, tool allowlists, and UI privileges. fileciteturn5file0L54-L71 fileciteturn2file0L44-L45

## Tool abstraction principles and long-term evolution

The right stable abstraction boundary is **not** “one tool per user-visible action.” The repo spec already points away from that approach: model-facing one-off composite tools are explicitly rejected, while composite helpers such as via patterns, fanout, stitching, or more elaborate routing helpers are expected to live in the SDK or composite library and lower into atomic operations. That is the correct long-term direction for Next Action as well. Stable tools should correspond to enduring editor invariants: observe state, query entities, mutate shadow state, generate candidates, validate, render, checkpoint, roll back, publish, reject, accept. Highly specific workflows should emerge from composition over those capability families rather than being enshrined as bespoke interfaces. fileciteturn2file0L57-L83 fileciteturn2file0L85-L95

A good KiSurf tool contract should usually return five things: a typed result, a compact semantic summary, the affected handles or region, validation or warning facts, and provenance metadata linking the result to session, step, checkpoint, and operation ids. Anthropic’s current guidance explicitly calls out clear tool namespaces, meaningful context in responses, and token-efficient outputs; OpenAI’s current APIs similarly emphasize typed schemas and structured outputs so the model is not forced to parse ambiguous textual blobs. That combination is ideal for an LLM-mediated EDA runtime, because it keeps the LLM focused on decisions instead of low-level parsing and lets new tools slot into the runtime without changing the core orchestration pattern. citeturn14view5turn14view1turn14view3

The long-term payoff of this design is extensibility without architectural drift. If KiSurf later adds a more capable differential-pair planner, a via-stitching generator, a room-aware placement solver, an EMI-oriented checker, or a richer form-filling inference engine, none of those should need a new “special agent.” They should register as tools that operate on the same session, shadow board, journal, preview, validation, and accept primitives. The LLM loop stays constant; only the available capabilities improve. That is exactly the kind of typed, auditable, governance-friendly separation that recent agent architecture surveys argue is emerging as the production-grade norm. citeturn14view8turn14view6

The final architecture I recommend is therefore straightforward to state. **Next Action should be one unified, LLM-mediated runtime profile on top of KiSurf’s existing session architecture.** The LLM owns semantic control: whether to suggest, what to try, how to interpret tool output, when to retry, when to give up, and when to publish. Deterministic algorithms own calculation: route geometry, placement candidates, validation, DRC, connectivity, rendering, and other editor-native computations. The runtime owns safety and product discipline: shadow isolation, checkpoints, rollback, provenance, invalidation, preview surfacing, and one-commit accept. That design best matches KiSurf’s existing branch, preserves “LLM as the brain,” avoids turning planners into hidden co-agents, and should scale cleanly as KiSurf grows from preview-first PCB workflows into a broader AI-native EDA system. fileciteturn2file0L147-L184 fileciteturn2file0L200-L262 fileciteturn3file0L151-L164 fileciteturn5file0L29-L52

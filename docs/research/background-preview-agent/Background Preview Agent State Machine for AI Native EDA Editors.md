# Background Preview Agent State Machine for AI Native EDA Editors

## Executive summary

KiSurfтАЩs own stated product loop is **тАЬSuggest, Accept, MaterializeтАЭ**: the system should observe the live design context, propose the next concrete action inside the actual workspace, let the engineer review it in place, and then materialize it after a lightweight confirmation similar to pressing **Tab** in a coding assistant. GitHubтАЩs research on Copilot points to the same core constraint from a different domain: developers reported that inline AI is valuable largely because it helps them **stay in flow** and **preserve mental effort**, which means interruption cost is a first-order design parameter, not a polish issue. For a background PCB/EDA agent, the central problem is therefore not only тАЬhow to generate suggestions,тАЭ but тАЬhow to manage suggestion lifetime under constant context drift without breaking the engineerтАЩs working rhythm.тАЭ юИАciteюИВturn11view0юИВturn27search0юИВturn27search18юИБ

The strongest pattern across IDEs, CAD, and EDA tools is this: **compute frequently, present selectively, preview transiently, commit only on explicit confirmation, and invalidate aggressively when the userтАЩs intent context changes**. VS Code and Copilot Next Edit Suggestions use a low-friction, location-aware interaction with collapsed display modes, directional indicators, and a two-step **Tab-to-navigate, Tab-to-accept** flow. CursorтАЩs Tab system goes even further, saying explicitly that suggestions should appear only when the model has **sufficient confidence**, because low-acceptance suggestions are distracting and disruptive. KiCad and Autodesk tools show the complementary graphics-side lesson: previews are best treated as **ephemeral overlays or preview objects** scoped to a tool or gesture lifetime, not as silent mutations of the live model. юИАciteюИВturn30view0юИВturn34view0юИВturn34view1юИВturn28view0юИВturn28view1юИВturn31view0юИВturn25view2юИВturn25view3юИБ

The recommended long-term design for KiSurf is therefore **not** a chat-like multi-suggestion inbox sprayed directly onto the board. It is a **single-primary-preview state machine** with explicit distinction between **HintOnly**, **Previewing**, **Accepted**, **Rejected**, **Expired**, and **Superseded**, driven by four independent context signals: **board revision**, **selection revision**, **view revision**, and **tool-state hash**. In practice, only one suggestion should occupy the canvas at a time; additional suggestions may exist, but only as passive grouped backlog entries, not simultaneous on-canvas overlays. юИАciteюИВturn28view0юИВturn30view0юИВturn34view3юИВturn31view0юИВturn29view1юИВturn33view0юИБ

## What existing systems teach

In IDEs, the leading pattern is no longer тАЬautocomplete only at the caret,тАЭ but **next-action prediction** with carefully constrained presentation. VS CodeтАЩs inline completion API says providers are invoked when the user stops typing, when inline completion is explicitly triggered, and when the user requests next or previous inline completions. The same API also says that if a normal completion item is selected, any inline preview must extend that same selected item and use the same range, and providers are requested again whenever the selected completion changes. That is a very strong engineering signal: inline AI suggestions are expected to be tightly bound to the userтАЩs current micro-context, and they are revalidated immediately when that micro-context changes. юИАciteюИВturn26view0юИБ

GitHub CopilotтАЩs next edit suggestions add three UI patterns that are directly relevant to KiSurf. First, suggestions can live **away from the current cursor position**, but the user is guided to them with a **directional gutter indicator**. Second, the product supports a **collapsed mode** that shows only the indicator until the user explicitly reveals the suggestion, specifically to reduce distraction. Third, the default acceptance model is **Tab to navigate, Tab again to accept**, and after one suggestion is accepted, **related** suggestions may appear automatically while **unrelated** ones remain hidden. This is almost a complete blueprint for PCB suggestion UX: directional anchoring, collapsed/hint state, two-stage acceptance, and chaining only for semantically related follow-up actions. юИАciteюИВturn30view0юИВturn34view0юИВturn34view1юИВturn34view2юИВturn34view3юИБ

Cursor contributes the most useful operational lesson. Cursor says its Tab model attempts a prediction on every typing or cursor-move action, **but only displays the suggestion if it has sufficient confidence**. It also states plainly that low accept rate means the system is showing too many incorrect suggestions, which is **distracting and disrupts the flow of coding**. Cursor further treats accept/reject outcomes as essential telemetry and uses them to improve the model and its display policy. For KiSurf, this means the right architecture is not тАЬwait to infer until we are sureтАЭ; it is тАЬinfer aggressively in the background, but gate publication with a high-confidence display policy and train against acceptance, rejection, expiry, and supersession outcomes.тАЭ юИАciteюИВturn28view0юИВturn28view1юИВturn28view2юИБ

Cursor also broadens the concept from тАЬinsert text hereтАЭ to **predict the next action across the workspace**, including predictions for **where to move next**. That matters because PCB editing, like code editing, often involves action sequences across different loci: select a component, inspect a net, zoom to a hotspot, place a via, then continue routing elsewhere. A background agent for KiSurf should therefore be conceived as a **next-action predictor with native overlay output**, not as a conversational advisor that happens to draw things. юИАciteюИВturn28view3юИВturn28view4юИБ

On the CAD/EDA side, KiCad already embodies the right graphics philosophy. The PCB editor exposes a **measurement tool** whose output is intentionally temporary and only shown while the tool is active. The interactive router has explicit modes such as **Highlight Collisions**, **Walk Around**, and **Shove**, and KiCadтАЩs documentation says the router prevents committing tracks into DRC-violating states in normal operation; for example, when dragging a footprint in collision-prone mode, a violating placement is rejected and the footprint returns to its original position. This is exactly the kind of тАЬtransient preview first, model commit only on a valid explicit actionтАЭ lifecycle a background AI preview should respect. юИАciteюИВturn38view0юИВturn38view3юИБ

At the rendering layer, KiCadтАЩs `KIGFX::VIEW` already separates durable scene content from transient preview content. It supports `MakeOverlay()`, `InitPreview()`, `AddToPreview()`, `ShowPreview()`, and `Hide()`; `AddToPreview()` can temporarily hide underlying items while overlaying preview items; and the view system also supports **display-only layers** that are rendered but excluded from hit testing. `VIEW_ITEM` additionally supports forced transparency for diff-like presentation. This means KiSurf does not need to invent a foreign preview mechanism for the background agent. The correct move is to build on the same preview/overlay vocabulary the editor already uses. юИАciteюИВturn15view0юИВturn29view0юИВturn29view1юИВturn29view2юИВturn29view3юИВturn29view4юИВturn33view0юИВturn33view2юИВturn33view3юИБ

KiCadтАЩs own overlay examples reinforce the recommendation. `CREEPAGE_OVERLAY` is documented as a **live creepage overlay shown while dragging items**; it has explicit `Start`, `Update`, and `Stop` methods; it draws through a single `VIEW_OVERLAY`; and it recomputes during the drag session rather than mutating the board. KiCadтАЩs class catalog also shows a dedicated family of preview items such as `POLYGON_ITEM`, `RULER_ITEM`, `SELECTION_AREA`, `SIMPLE_OVERLAY_ITEM`, `SNAP_INDICATOR`, and `TWO_POINT_ASSISTANT`, all of which are purpose-built for temporary, visual, interaction-scoped feedback. The precedent is clear: a background AI preview should behave like a disciplined interactive tool, not like a hidden model editor. юИАciteюИВturn31view0юИВturn36view0юИВturn36view1юИВturn36view2юИВturn36view3юИБ

Autodesk Fusion and AutoCAD point in the same direction. FusionтАЩs API guidance says geometry changes for previews belong in `executePreview`, not in `inputChanged`, and AutodeskтАЩs official `CommandEventArgs.isValidResult` property exists specifically to tell Fusion that an existing preview result can be reused as the final result on confirmation. AutoCADтАЩs command preview is explicitly described as an **interactive preview displayed before the command is completed**, and the feature is user-toggleable. Together, these are strong cross-domain signals that preview lifetimes should be **explicit, user-reviewable, and separate from final commit**, while aggressiveness should remain controllable by the user. юИАciteюИВturn23search1юИВturn25view2юИВturn25view3юИВturn25view4юИБ

## Recommended state model

The most stable design is a **single suggestion object** with three layers of validity and one explicit UI state. The three validity layers should be treated separately:

A suggestion is **render-valid** if KiSurf can still draw it in the right place. It is **semantic-valid** if it still matches what the engineer appears to be doing. It is **accept-valid** if the exact preview can still be promoted into the standard typed-operation acceptance path without recomputation. This distinction matters because a suggestion can remain semantically useful even after a viewport pan, while becoming non-renderable if dependent objects were deleted, and it can remain renderable while becoming non-acceptable if the board base revision has changed. This layering is the natural generalization of IDE inline-preview constraints and CAD/EDA transient preview semantics. юИАciteюИВturn26view0юИВturn34view1юИВturn31view0юИВturn25view2юИБ

The context signature for each suggestion should contain, at minimum, four revision domains. **Board revision** should advance on each accepted live-board mutation, carrying a touched-object set and dirty bounding box. **Selection revision** should advance when the selected set or anchor object changes. **View revision** should advance on pan, zoom, visible-layer changes, or other viewport-affecting adjustments. **Tool-state hash** should encode the active tool, submode, active layer, routing mode, drag state, and other modal interaction flags. This is not how KiCad names these concepts today, but it is the right logical decomposition because it maps directly onto the different ways existing systems revalidate suggestions: text-selection movement in IDEs, tool lifetime in KiCad overlays, and explicit preview-phase versus commit-phase separation in CAD command systems. юИАciteюИВturn26view0юИВturn31view0юИВturn38view0юИВturn38view3юИБ

The UI-facing state machine should be deliberately small:

- **Pending**: the agent has produced or is producing a candidate suggestion for a specific context signature, but nothing is shown yet. Validation and confidence scoring happen here.  
- **HintOnly**: the suggestion is valid enough to surface, but the system shows only a lightweight indicator, such as a canvas badge, edge-arrow, glow target, or side-strip card. This mirrors CopilotтАЩs collapsed mode. юИАciteюИВturn30view0юИВturn34view0юИВturn34view3юИБ
- **Previewing**: the native canvas overlay is visible, scoped to the suggestionтАЩs current context, and the acceptance affordance is armed. This is analogous to KiCadтАЩs transient preview items and live overlays. юИАciteюИВturn29view1юИВturn31view0юИВturn36view0юИБ
- **Accepted**: the user explicitly confirms the suggestion, and the preview hands off to KiSurfтАЩs typed-op accept pipeline. In Fusion terms, this is the moment where a valid preview becomes the final result. юИАciteюИВturn25view2юИБ
- **Rejected**: the user explicitly dismisses it, or implicitly rejects it by acting against it. This state should feed suppression and learning. CursorтАЩs design makes accept/reject telemetry central for exactly this reason. юИАciteюИВturn28view2юИБ
- **Expired**: the suggestion was not rejected by the user, but its context became invalid. This must be distinct from rejection because the product needs to learn тАЬstaleтАЭ separately from тАЬbad.тАЭ The need for immediate revalidation on context change is strongly suggested by VS CodeтАЩs inline completion contract and by KiCadтАЩs interaction-scoped overlays. юИАciteюИВturn26view0юИВturn31view0юИБ
- **Superseded**: a newer suggestion for the same context group replaced it. This should also be separate, because тАЬold suggestion replaced by better oneтАЭ is not the same as тАЬuser disliked it.тАЭ CopilotтАЩs тАЬrelated suggestions continue, unrelated suggestions stay hiddenтАЭ is the clearest product analogue. юИАciteюИВturn30view0юИБ

The most important practical rule is that **only one suggestion should be in `Previewing` on the canvas at a time**. Background AI suggestions are only helpful when they feel like the **next reasonable move**, not when they compete for attention. CursorтАЩs emphasis on noisy suggestions lowering accept rate, together with CopilotтАЩs collapsed-mode and chained-related-suggestions pattern, strongly supports a **single-primary-preview** policy rather than a parallel preview swarm. юИАciteюИВturn28view0юИВturn34view3юИВturn30view0юИБ

## Invalidation and supersession strategy

A useful way to make invalidation predictable is to classify suggestions into three scopes: **selection-scoped**, **viewport-local**, and **workspace-global**. Selection-scoped suggestions are things like тАЬcontinue this route,тАЭ тАЬdrop stitching vias around this selected zone edge,тАЭ or тАЬnudge this footprint to relieve congestion.тАЭ These should expire immediately on selection revision changes or incompatible tool-state changes. Viewport-local suggestions are things like тАЬrecommended fanout in this visible congestion pocketтАЭ; these can survive small selection changes but should collapse when the user moves far away. Workspace-global suggestions are strategic but non-urgent, such as тАЬthis power net wants another decoupler pathтАЭ or тАЬthese two nets likely need tighter return-path treatmentтАЭ; they should generally not auto-preview on the canvas at all and should live as passive hints. This tiering is an inference from how Copilot distinguishes related and unrelated next edits, and from how KiCad preview objects are tied to tool-local interaction. юИАciteюИВturn30view0юИВturn31view0юИВturn36view0юИБ

**Board revision** should be the strongest invalidator. If a user edit touches the suggestionтАЩs dependent object IDs or intersects its dependency region, the suggestion should move to **Expired** immediately. If the board changed elsewhere, the system should attempt a fast revalidation; if the suggestion still resolves cleanly, it may remain in `HintOnly` or `Previewing`. This is important because engineers often make local edits while visually inspecting adjacent regions, and overly eager invalidation would create flicker and destroy trust. KiCadтАЩs live overlay pattern suggests exactly this kind of local recompute loop rather than global teardown for every frame. юИАciteюИВturn31view0юИВturn29view1юИБ

**Selection revision** should usually be a hard invalidator for selection-scoped suggestions and a soft invalidator for viewport-local ones. In the MVP, I would keep this simple: if the selected object set changes and the suggestion depended on selection, expire it. Later, KiSurf can support compatibility classes such as тАЬsame root footprint groupтАЭ or тАЬsame net family,тАЭ but the first version should prefer correctness and predictability over clever persistence. That matches the conservative invalidation behavior implied by IDE inline completion contracts that refresh whenever selected completion context changes. юИАciteюИВturn26view0юИБ

**View revision** should almost never destroy a suggestion by itself. Instead, it should drive **presentation downgrade**. If the user pans away, zooms out too far, or hides relevant layers, the suggestion should usually go from `Previewing` to `HintOnly`, with a directional marker or unobtrusive chip indicating where it lives. This is exactly what Copilot NES does when the next suggestion is outside the current view: the user is shown where it is, not forcibly yanked there. In KiSurf, view changes should therefore lead to **collapse**, not immediate expiry, unless the suggestion genuinely cannot be rendered meaningfully anymore. юИАciteюИВturn30view0юИВturn34view1юИБ

**Tool-state hash** should govern intent compatibility. If the user switches from routing to dimensioning, starts a drag gesture, enters a modal dialog, or focuses the chat panel or a text field, a background PCB construction preview should lose precedence immediately. For compatible tool transitionsтАФsuch as moving from one routing submode to anotherтАФKiSurf can revalidate and keep the suggestion if it still makes sense. KiCadтАЩs measurement tool, router modes, and creepage overlay all demonstrate that transient graphics are most stable when they are **owned by one interaction mode at a time**. юИАciteюИВturn38view0юИВturn38view3юИВturn31view0юИБ

For **multiple simultaneous suggestions**, the right rule is: one visible preview, a tiny passive backlog, grouping by context. Concretely, I would keep **one active canvas suggestion overall**, plus at most **three passive backlog entries** grouped by context family such as *routing follow-up*, *placement nudge*, and *cleanup/review*. If a new suggestion arrives in the same context group, the old one becomes **Superseded**. If it is in a different group and lower salience, it goes to backlog without painting the canvas. This mirrors CopilotтАЩs related-suggestion chaining and prevents the UI from becoming a recommendation carousel. юИАciteюИВturn30view0юИВturn34view3юИВturn28view0юИБ

## Throttling and publish policy

A background preview agent should be **event-driven internally but rate-limited externally**. CursorтАЩs model tries to predict on every action, which is sensible, but its own explanation only validates publishing when confidence is high enough to avoid noise. KiSurf should adopt the same split: observe everything, infer often, publish rarely. In practice, pointer-move, pan, and zoom events should be coalesced behind a short trailing debounce; selection and tool changes should cancel in-flight jobs immediately and spawn a fresh generation; and no new visual suggestion should publish during an active drag unless it is explicitly in the same assistance class as the current tool. юИАciteюИВturn28view0юИВturn28view1юИВturn31view0юИБ

The publish rule should be based on a **preview eligibility score**, not just model confidence. I recommend combining at least five factors: predicted acceptance probability, semantic stability probability, visual inspectability, estimated disruption cost, and suggestion radius. Small, local, high-confidence, immediately inspectable edits should be allowed to auto-preview. Medium-confidence or off-screen suggestions should surface as `HintOnly`. Low-confidence suggestions, repeated near-rejections, or suggestions generated during rapid user activity should remain completely silent. AutoCADтАЩs preview toggle and CopilotтАЩs collapsed mode both support making display aggressiveness a user-facing preference rather than a hardwired product doctrine. юИАciteюИВturn25view3юИВturn25view4юИВturn34view3юИВturn28view0юИБ

The product should also require a brief **stability dwell** before painting. A good heuristic is: only show a suggestion if the context signature remains semantically stable across a short window or across two consecutive inference generations. This prevents oscillation when the user is still moving rapidly through intermediate states. The same general principle underlies CursorтАЩs high-accept-rate focus and the interaction-scoped nature of KiCad overlays. юИАciteюИВturn28view0юИВturn31view0юИБ

Telemetry should treat the following as distinct outcomes: **accepted**, **explicitly rejected**, **expired by board drift**, **expired by tool change**, **collapsed then ignored**, **superseded**, and **manually performed equivalent edit**. CursorтАЩs public description makes accept/reject outcomes central to product improvement, and KiSurf will need even richer labels because тАЬstale during fast editingтАЭ is not the same product problem as тАЬbad suggestion.тАЭ юИАciteюИВturn28view2юИБ

## Accept interaction and mis-trigger prevention

For keyboard acceptance, the safest design is a **two-stage accept model** exactly analogous to Copilot NES. When a suggestion is in `HintOnly`, the first **Tab** should focus or reveal the preview; the second **Tab** should accept it. If the preview is already visible and focused, one **Tab** may accept it. If focus is in chat, a text field, a modal dialog, or an incompatible editor tool, **Tab must keep its normal native meaning**. MicrosoftтАЩs own documentation for NES explicitly uses the тАЬTab to navigate, Tab to acceptтАЭ pattern and even warns about keybinding conflicts in editor configurations, which is a useful reminder that global accept bindings need a very explicit arming condition. юИАciteюИВturn30view0юИВturn34view2юИБ

For pointer and touch interaction, I would not recommend a global тАЬtap anywhere to acceptтАЭ behavior. In a PCB editor, taps and clicks already mean selection, point placement, drag start, and routing confirmation. The safer pattern is: clicking the preview itself focuses it; clicking a visible **Accept** chip or pressing the armed hotkey confirms it; `Esc` dismisses it; and panning away collapses it back to `HintOnly`. CAD command previews and KiCad transient tools are all built around explicit mode ownership, not ambient accidental confirmation. юИАciteюИВturn25view3юИВturn31view0юИВturn38view0юИБ

A subtle but important behavior from MicrosoftтАЩs collapsed NES should be copied directly: after accepting one suggestion, **related** follow-up suggestions may appear immediately, but **unrelated** ones should remain collapsed. In KiSurf terms, if the user accepts тАЬfinish stitching along this selected edge,тАЭ then showing тАЬadd the last two corner viasтАЭ immediately is reasonable. But after that, an unrelated тАЬtry rotating U12тАЭ suggestion should stay passive. This sequencing makes the system feel coherent instead of opportunistic. юИАciteюИВturn30view0юИБ

## KiSurf-specific implementation recommendation

For KiSurf, the recommended design is a dedicated **Background Suggestion Manager** that sits above inference and below the existing typed preview/accept path. Its responsibilities should be to maintain the current context signature, track suggestion generations, classify validity, arbitrate visibility, and drive native preview rendering through KiCadтАЩs existing `KIGFX::VIEW` preview and overlay facilities. The manager should not own board mutation itself; once a suggestion is accepted, it should hand off to the same typed-op preview/commit machinery used by the chat agent and other AI-assisted edit paths. This respects KiSurfтАЩs own тАЬsuggest, accept, materializeтАЭ philosophy while keeping background suggestions inside the same auditable edit architecture. юИАciteюИВturn11view0юИВturn29view0юИВturn29view1юИВturn29view2юИВturn29view3юИБ

On the graphics side, I would implement two visual channels. The first is a **collapsed channel** using a lightweight overlay or dedicated display-only AI layer for arrows, anchor glows, target circles, or miniature badges. Because `KIGFX::VIEW` supports display-only layers that are rendered but excluded from hit testing, this is a natural fit for passive hints. The second is a **focused preview channel** using `InitPreview()` and `AddToPreview()` for fully inspectable temporary PCB geometry. If KiSurf wants diff-like emphasis, `VIEW_ITEM`тАЩs forced transparency can be used to visually distinguish the live board from the proposed edit. юИАciteюИВturn33view0юИВturn29view1юИВturn33view2юИВturn33view3юИБ

The background agent should also inherit a principle from KiCadтАЩs creepage and measurement tooling: **preview lifetime should be explicit and short**. `Start тЖТ Update тЖТ Stop` is a much better mental model than тАЬpreview objects live until the model says otherwise.тАЭ For KiSurf, that means each suggestion should have an ownership lifetime tied to a context generation, not merely to elapsed time. If the next generation invalidates it, the overlay should be removed immediately and cleanly. юИАciteюИВturn31view0юИВturn38view0юИБ

The most important product decision is scope. A **background preview agent should optimize for micro-actions**, not long autonomous construction sequences. In code editors, ghost text and next edit suggestions are powerful precisely because they represent the *next likely move*, not an entire multi-minute refactor executed invisibly. KiSurfтАЩs background agent should therefore focus on local placement nudges, routing continuations, fans, stitch suggestions, instant cleanup candidates, and constraint-aware visual hints. Multi-step tasks with branching intent should escalate into the chat agent or a richer explicit workflow. That division best matches both the KiSurf vision and the accumulated interaction lessons from Copilot, Cursor, and CAD preview systems. юИАciteюИВturn11view0юИВturn30view0юИВturn28view3юИВturn25view4юИБ

The final recommendation is concise: **one active preview, explicit context signatures, aggressive invalidation on semantic drift, collapse instead of expire on pure viewport drift, two-stage Tab acceptance, related-suggestion chaining only, and native transient rendering through KiCadтАЩs existing preview/overlay stack**. That gives KiSurf a background agent that behaves less like an intrusive chatbot and more like a disciplined, high-signal co-editor embedded in the canvas itself. юИАciteюИВturn11view0юИВturn34view3юИВturn28view0юИВturn31view0юИВturn29view2юИБ

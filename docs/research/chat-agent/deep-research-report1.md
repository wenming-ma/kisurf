# Recommended Architecture for an AI-Native PCB Editor Agent Execution Layer

## Bottom line

The best long-term design is a **hybrid, narrow-waist architecture**: keep **all board mutation** behind a **stable, typed atomic operation layer**; add a **typed execution IR** above it for variables, loops, conditions, geometry, checkpoints, and rollback; run that IR inside a **step-wise preview session** backed by the **native KiCad renderer and transaction system**; and allow multiple authoring frontends on top of the same core, including **JSON AST / typed JSON**, later **Python/JS/Lua frontends**, and eventually **WASM components** for stronger isolation and multi-language support. This recommendation matches what the KiCad team is already doing at the API boundary with a stable Protobuf-based IPC API, what KiCad’s own tool framework does with `BOARD_COMMIT`, what Qt/AutoCAD/FreeCAD do with command/transaction-based undo, and what Fusion/VS Code do with preview-first or review-first workflows. citeturn2view1turn15view0turn20view0turn5search4turn5search13turn10search0turn10search1turn13search2turn23view1

In practical terms, the canonical pipeline should be:

```text
Agent
  -> plan
  -> typed execution IR
  -> per-step operation bundle
  -> shadow board session
  -> native preview render + board-state query + DRC/rule feedback
  -> revise / rollback / continue
  -> final accept
  -> single KiCad commit into undo stack
```

That gives you the properties you explicitly want: arbitrary composition from stable atomics, step execution, intermediate visual inspection, checkpoint/rollback, user-visible progress, final accept gating, and a backend that can survive multiple future frontends without being rewritten. citeturn15view0turn22search0turn23view1turn10search1turn10search16

## What existing systems already prove

KiCad itself already points toward the right architectural direction. The old SWIG Python bindings are explicitly documented as a **raw binding layer** that is **unstable** across major versions, while the newer IPC API is documented as a **stable, language-agnostic interface** based on **Protocol Buffers** and **NNG**, with additive versioning rules that forbid changing the meaning of existing fields. KiCad also launches IPC plugins as **standalone processes**, not in-process extensions, and can log every API request/response at the protobuf message level. Those are exactly the traits you want for a durable AI-facing substrate: stable schema evolution, language neutrality, process isolation, and auditable traffic. citeturn21view2turn2view1turn2view2turn15view1

KiCad’s editor-side mutation model also already has the right shape. In the GAL tool framework, `COMMIT` / `BOARD_COMMIT` is used to collect **modifications, additions, and removals** into a **single undo/redo action**; you mark items with `Modify`, `Add`, or `Remove`, then finalize with `Push`, and if you destruct without pushing, the commit is aborted. That is almost exactly the semantic boundary an AI session should target on final accept: many low-level changes grouped into one coherent editor commit. KiCad’s rendering stack also contains dedicated native preview infrastructure, including `KIGFX::PREVIEW::DRAW_CONTEXT` and multiple preview item types for in-progress geometry and overlays, which is strong evidence that the editor already has native concepts for temporary visual state rather than only final committed items. citeturn15view0turn22search0turn22search1

Other CAD systems reinforce the same pattern. Blender exposes **low-level, chainable mesh operators** through `bmesh.ops`, documents that operators which modify data must participate in **undo**, and supports **macro operators** and grouped undo. FreeCAD exposes explicit document transactions with `openTransaction`, `commitTransaction`, `abortTransaction`, pending-transaction checks, and undo stack sizing. AutoCAD’s .NET API describes transactions as a way to group multiple object operations into a single operation, supports **nested transactions**, and exposes **commit** or **abort/rollback** semantics. Qt’s undo framework is literally described as an implementation of the **Command pattern**, with **command macros**, **child commands**, a **clean state**, and stack index navigation. Mature editors do not scale by adding one more composite API per user task; they scale by combining **small operations + transactions + undo/redo + preview**. citeturn3search2turn3search5turn3search23turn20view0turn20view1turn5search4turn5search8turn5search13turn10search0turn10search1turn10search2turn10search4turn10search16

Fusion 360 is particularly relevant for the preview question. Autodesk documents `executePreview` plus `isValidResult`, and Autodesk’s own expert guidance explains that preview work can be created in `executePreview`, automatically discarded when inputs change, and optionally reused as the final result when marked valid. Fusion also provides **custom graphics** for extra temporary or explanatory overlays. That is the closest mainstream CAD precedent for what you want: a real modeling loop where the system renders an actual intermediate result, lets the user inspect it, and only finalizes when accepted. citeturn2view4turn13search0turn13search7turn12search1

Modern AI-enabled editors show the same control-loop pattern at a higher level. VS Code’s tool API is built around tools with **descriptions and input schemas**, custom editors are required to emit explicit edit events with **undo/redo handlers**, and Copilot’s agent workflows emphasize **proposed edits, review, undo, and iteration based on runtime feedback**. Copilot Edits surfaces changes in-place and lets users **accept or discard** them; GitHub’s documentation also warns that autonomous tools can be used without approval if you configure them that way, which is exactly why a PCB editor should keep mutation behind an editor-owned preview/accept gate rather than handing raw authority straight to the model. citeturn23view2turn23view1turn11search12turn11search15turn17search11

## Comparative assessment of execution models

The option that should be **canonical** is **JSON AST / typed IR**, not because JSON is elegant, but because a typed intermediate form is the easiest place to enforce **schema validation, deterministic semantics, audit logging, quota checks, geometry/rule validation, replay, and cross-language compatibility**. KiCad’s own stable IPC boundary is Protobuf-based and versioned additively, while WIT in the Wasm component model and input schemas in the VS Code tool API both show the same underlying principle: stable systems scale best when the boundary is **typed, documented, and composable**. My recommendation is therefore: define the execution model once as a **schema-first IR**, serialize it as **Protobuf internally** and as **JSON for model generation/debugging**. citeturn2view1turn15view1turn7search2turn7search10turn17search2turn23view2

**Python/Lua/JS sandboxes** are valuable, but they should be treated as **frontends**, not as the canonical mutation engine. Python’s own documentation warns that `eval` executes arbitrary code and is unsafe on untrusted input. Node’s official docs are even more explicit: `node:vm` is **not a security mechanism** and should not be used for untrusted code. Lua is excellent as an embeddable language, and its docs make clear that chunks can be run with alternate environments, but by default free names resolve through the global environment and standard libraries are loaded there. In other words: these languages are good for ergonomics and ecosystem adoption, but if the model itself is generating code, the host must either harden them aggressively or, better yet, make them compile or bind into the same typed IR and capability-limited host API rather than directly mutating the board. citeturn8search1turn8search0turn18search0turn18search5

**WebAssembly** is the best long-term answer for **trusted extension isolation** and **future multi-language support**, but it is not the fastest thing to ship as the first user-facing authoring mode. The Wasm component model is explicitly designed for interoperable WebAssembly components; WIT defines interface contracts rather than behavior; worlds define strict import/export boundaries; and Wasmtime provides production-grade mechanisms for **resource limits**, **instance/table/memory limits**, and **safe interruptibility** through epoch deadlines or fuel. That makes Wasm an excellent boundary for third-party “script engines” or advanced user plugins once the core IR is settled. It is a great **runtime container**, but the winning strategy is still: **typed IR first, Wasm as one frontend/runtime boundary on top of it**. citeturn7search6turn7search2turn7search12turn7search1turn7search9turn7search11turn7search0

**Workflow graph / node graph** systems are useful, but they should be a **secondary frontend**, mainly for reusable human-authored templates and visual macros. Blender’s node groups are explicitly described as reusable, composable, and parameterizable, and Unity’s visual scripting docs describe graphs, variables, and formula nodes. Those are strong precedents for a human-oriented “playbook builder.” But for an LLM, node graphs are usually too verbose, too spatial, and too awkward to patch step-by-step in text. They are better as a UI that **compiles to the same IR**, not as the primary interface the model emits. citeturn9search3turn9search16turn9search6turn9search2turn9search22

**LLM tool-calling plus operation bundle** is useful but insufficient by itself. Tool-calling is a very good outer orchestration layer—VS Code’s tool APIs and MCP reflect that—but a pure tool-call loop does not naturally give you variables, references to created objects, geometry expressions, loop bodies, checkpoint semantics, or a durable replay format. It is therefore best used as the **outer control protocol**: the agent chooses actions like “compile_and_step,” “rollback_to_checkpoint,” “query_board_state,” or “render_preview_frame,” while the executable content inside each step is a typed script/IR that lowers to atomic operations. citeturn23view2turn17search1turn17search4turn17search18

So the comparative conclusion is clear: the **best architecture is hybrid**. Make **typed IR + atomic op executor** the narrow waist. Let **tool-calling** orchestrate it. Allow **Python/JS/Lua** and **workflow graphs** as optional frontends. Use **WASM** as the long-term extension sandbox. That gives you one stable backend and many replaceable authoring surfaces. citeturn2view1turn15view1turn7search6turn23view2

## Recommended long-term architecture

The core design should have **three execution layers**, each with a different purpose. The bottom layer is the **atomic board API**. This layer should expose only stable, typed mutations and queries: `create_via`, `create_track_segment`, `create_zone_outline`, `move_item`, `delete_item`, `set_net`, `set_rule_attrs`, `query_board_state`, `query_selection`, `run_drc`, `render_frame`, and similar operations. This is the layer that gets versioned like KiCad’s IPC protobufs: additive fields only, no semantic rewrites of existing messages. That is the only layer allowed to mutate board state. citeturn2view1turn15view1turn15view0

Above that, define a **typed execution IR** that is richer than a mere operation list. It should support `let` bindings, bounded `for` loops, `if` conditions, pure expressions, geometry/math helpers, collection transforms, references to prior results, temporary object handles, explicit checkpoints, rollback statements, assertions, and step labels. Importantly, this layer should be **deterministic** and **host-capability-limited**: no file I/O, no network, no wall-clock time, no random by default. Conceptually, think of it as “a tiny deterministic geometry-and-edit language that compiles to atomic board ops.” The reason to keep an IR instead of asking the model to emit raw atomics is the same reason CAD and editor systems use macros and transactions: humans and agents need a middle layer where logic can be expressed clearly before it is lowered into small edits. citeturn10search0turn10search1turn10search4turn15view0

At the top, support multiple **authoring frontends** that all target that same IR. The first frontend should be **typed JSON / JSON AST** because it is the easiest for LLM generation, validation, and diffing. The second wave can add a **Pythonic DSL**, possibly also a JS or Lua DSL, but these should map into the same host functions and produce the same IR/logging pathway. Later, if you want a strong plugin story, add a **WASM component frontend** with WIT-defined interfaces, so Rust, C#, TinyGo, or other languages can participate without changing the editor core. The value of the component model here is not fashion; it is the ability to add languages later while keeping one stable contract at the boundary. citeturn7search6turn7search2turn7search12turn17search2turn23view2

The editor-side executor should run in a **preview session with a shadow board**. During the session, the real PCB is not yet committed. Instead, each step of the IR produces a **per-step operation bundle** that is validated and applied to a shadow board, then rendered by the **native KiCad renderer**, then observed by both user and agent. This is the crucial difference from “generate one script and run it once”: the agent should work like an engineer, meaning the system must support a repeated **act → inspect → revise** loop, just as Fusion uses preview results and modern coding agents iterate based on compiler/test output. citeturn13search0turn13search7turn12search1turn11search12turn17search16

The observation API of that session should expose **three synchronized views of state**. First, a **native preview frame** or region crop from the live editor canvas. Second, a **structured board-state snapshot** with exact geometry, nets, metadata, rules, and handles. Third, a **validation channel** containing DRC/rule findings, geometry warnings, and executor diagnostics. KiCad already has robust design-rule concepts and preview drawing primitives; the missing piece is to wrap them into an agent-facing observation contract. Without this observation layer, you will never get robust self-correction from the model. citeturn19search1turn19search3turn22search0turn22search1

## Preview, checkpoint, rollback, and accept

The system should treat a session as a **timeline of step commits inside preview**, not as a single monolithic transaction. Each step should have a label, a generated operation bundle, a validation result, a rendered preview, and an after-state snapshot. When the agent says “place 32 vias around this ring,” that should be one named step in the session timeline even if it lowers into dozens of atomic `create_via` and `set_property` operations. This mirrors how Qt macros and command trees keep internal child commands while presenting a cleaner top-level unit to the user. citeturn10search1turn10search2turn10search4turn10search16

Checkpointing should be **first-class**. The executor should allow explicit `checkpoint("name")` boundaries and retain at least a log index plus lightweight state snapshot for each one. Rollback should default to **step-atomic behavior**: if a step fails validation or execution, revert that entire step and keep the previous checkpoint intact. More advanced scripts can opt into exploratory behavior, but the default should be safe and predictable. FreeCAD, AutoCAD, Qt, and KiCad all show that transaction boundaries are the right place to define rollback semantics; the same idea should be used here inside the AI preview session. citeturn20view0turn20view1turn5search8turn5search13turn10search1turn15view0

Preview should be **render-first, list-second**. The primary thing shown to the user should be the actual board preview in the native canvas, because that is where spacing, position, polarity, clearance, and unintended geometry are easiest to catch. A semantic step list should sit beside it, and the atomic operation list should be available as an expandable audit/debug panel rather than the default UI. This is the best compromise between human usability and full traceability. Fusion’s preview model and VS Code’s edit-review model both point in this direction: users inspect the actual result, but the system still preserves a structured representation of the edits underneath. citeturn13search0turn13search7turn11search15turn23view1

Accept should not merely “bless” an opaque preview bitmap. It should finalize a **deterministic diff**. The cleanest design is: keep an **operation log as the source of truth**, use the shadow board for exact preview and intermediate checks, and at Accept time either **replay the validated op log** or apply a computed diff into the real board via one top-level `BOARD_COMMIT`. That produces the user-visible outcome you want: after Accept, the changes become native KiCad objects and enter the editor’s undo stack as one coherent AI edit. KiCad’s own `BOARD_COMMIT` and Qt’s macro-command model are the right precedents here. citeturn15view0turn10search0turn10search1turn10search2

Reject, Undo, and Replay should be different things. **Reject** discards the preview session entirely and leaves the real board untouched. **Undo** is the ordinary post-accept editor undo, ideally one top-level “AI session” entry in the KiCad stack. **Replay** is an internal engineering/debugging function that re-applies the step log from a checkpoint or from the beginning to reproduce a session. Qt’s `setIndex()`/undo stack model and VS Code’s explicit undo/redo edit events show the value of having a replayable edit history that is not just a side effect of pixels changing on screen. citeturn10search1turn10search16turn23view0turn23view1

## Safety, determinism, and auditability

Your safety boundary should be uncompromising: **no board mutation except typed atomic operations owned by the editor**. That single rule is more important than the choice of scripting language. Once every mutation funnels through a typed op layer, you can add checks that are impossible to guarantee in a free-form runtime: schema/type validation, coordinate normalization, layer legality, existing-object reference validation, per-step object count limits, per-session mutation budgets, copper-area change budgets, board-boundary checks, and DRC/rule checks before accept. KiCad’s rule/DRC system gives you the domain validation hooks, and KiCad’s protobuf versioning discipline gives you the model for keeping that layer stable over time. citeturn19search1turn19search3turn19search8turn15view1

If you support general-purpose scripting frontends, do not trust their built-in “sandbox” claims. Node’s official docs say the vm module is not a security mechanism. Python warns that `eval` executes arbitrary code. Lua is embeddable and more host-shaped, but by default uses a global environment unless you deliberately constrain it. So if you expose Python/JS/Lua at all, they should run **out of process** or in **WASM**, with **no filesystem/network imports**, and ideally with their only authority being a host object that produces typed IR or requests editor-owned operations. KiCad’s own IPC plugin model—standalone processes communicating with the editor over a stable API—is a better precedent than in-process arbitrary scripting. citeturn8search0turn8search1turn18search0turn18search5turn2view2

For resource control, combine **static limits** with **runtime caps**. The Wasmtime docs provide exactly the kind of mechanisms worth copying conceptually: store-level resource limiters, memory/table/instance limits, and safe deadline interruption. JavaScript worker threads also expose resource limits at the worker level. In your system, the equivalent should be a maximum step count, maximum emitted operations per step, maximum wall time per step, and maximum preview diffs before the agent must ask for a new checkpoint or operator confirmation. citeturn7search1turn7search9turn7search11turn7search0turn8search2

Auditability should cover both the AI layer and the editor layer. KiCad already supports protobuf-level API logging. Extend that idea so that every AI session stores: the plan, the generated IR, each lowered operation bundle, validator/DRC output, checkpoint metadata, rendered frame references, and final acceptance metadata. This is useful not only for debugging and user trust, but also for replay, regression testing, and eventual offline evaluation of agent reliability. citeturn2view2turn23view1turn10search16

## Roadmap and layering

The **short-term shippable design** is: keep using KiCad’s stable IPC direction and editor transaction model, define a **minimal typed execution IR** in Protobuf with a JSON serialization, implement a **preview session manager** with checkpoints on a shadow board, and let the model emit that JSON IR directly. Do not start with full Python execution. Start with a small but expressive IR: variables, bounded loops, if/else, pure geometry helpers, object handles, queries, checkpoints, rollback, and atomic edit ops. That will already cover the ring-zone / ring-via workflow you described without locking you into a brittle language runtime too early. citeturn2view1turn2view2turn15view0turn15view1

The **medium-term evolution** is to add **ergonomic frontends** without changing the backend. That means a Pythonic DSL for power users, maybe a JS/Lua frontend if your team prefers those ecosystems, and a visual workflow/node editor for reusable templates or parameterized macros. All of them should compile or bind into the same IR and step runner. At this phase, you should also add richer observation tools: crop render, multi-scale render, diff overlays, semantic changed-object lists, and fast DRC-lite checks after each step plus full DRC on demand. The point is that frontends multiply, but the execution core stays singular. citeturn9search3turn9search6turn9search22turn19search1turn19search3

The **long-term AI-native architecture** is a full **agent workbench** inside the PCB editor: the model plans, emits or revises typed IR, executes one preview step at a time, reads the resulting render/state/DRC feedback, rolls back when needed, and only commits when the user accepts. At that point, a **WASM extension boundary** becomes highly attractive for external “skills,” because WIT/component-model interfaces let you add languages and specialized components while keeping a strong, typed, capability-limited boundary. This is the right moment to add reusable third-party automation modules and team-specific macro packs. citeturn7search6turn7search2turn7search12turn17search1turn17search21

You should **not** define a separate interface for every composite action. Existing CAD/editor systems show why: composition space grows faster than API surface can be maintained, while stable typed boundaries evolve additively and let higher-level macros sit above them. KiCad’s protobuf rules explicitly favor additive evolution of stable messages; Blender’s BMesh operators are chainable; Qt supports macro commands with child commands; transaction systems in KiCad, FreeCAD, and AutoCAD group arbitrary low-level edits into higher-level units. The right abstraction is therefore not “one interface per task,” but “one stable mutation substrate under many changing tasks.” citeturn15view1turn3search2turn10search0turn10search4turn15view0turn20view0turn5search4

The cleanest final layering is:

```text
Atomic operations
  Stable, typed, versioned, editor-owned, the only mutation path

Composite libraries
  Reusable domain helpers like ring-via placement, serpentine generators, fanout recipes
  These are not privileged; they lower to atomic ops

Script layer
  Typed execution IR with variables, loops, conditions, checkpoints, rollback, assertions

Agent layer
  Planning, step scheduling, observation use, self-correction, human-facing explanations
```

That layering is what gives you long-term stability. The atomic layer changes slowly. Composite libraries evolve with product needs. The script layer grows in expressiveness. The agent layer changes fastest with model capabilities. By keeping those rates of change separate, you avoid the trap of rewriting the foundation every time the AI interaction model improves. citeturn2view1turn15view1turn15view0turn23view2turn17search16

## Bottom line

The strongest long-term architecture is **not** ŌĆ£Python directly driving KiCad internals,ŌĆØ and it is also **not** a pure external-plugin architecture. ## Comparing process models and transport choices

A fully **in-process embedded Python** runtime is attractive only on latency grounds. It gives direct callback convenience and avoids serialization overhead, but it is the worst fit for the operational properties you asked for: runaway code is hard to interrupt safely, state contamination is easy, native crashes or bad extension modules can take down the editor, and the temptation to leak raw board pointers into Python becomes very strong. KiCadŌĆÖs own API direction and VS CodeŌĆÖs extension-host design both point the other way: keep third-party or volatile execution **out of the UI/core process** so misbehaving code does not compromise editor responsiveness or stability. ŅłĆciteŅłéturn1view1Ņłéturn1view2Ņłéturn33view0Ņłü

A **long-lived global Python service** is better than in-process embedding, but it is still the wrong default for interactive PCB editing sessions. It makes startup cheaper, but it introduces cross-project contamination, state bleed between Agent tasks, package/version drift inside a resident interpreter, and more complicated recovery when a worker gets poisoned. Those tradeoffs are acceptable in some server products, but they are a poor match for deterministic, auditable editing sessions that must stay aligned with a specific board hash, viewport, selection state, and preview journal. The LSP pattern is useful here conceptuallyŌĆöeditor and language intelligence separated by protocolŌĆöbut your session state is much richer and more document-coupled than a traditional language server. ŅłĆciteŅłéturn33view1Ņłéturn1view2Ņłü

A **worker pool** is useful, but only for the right jobs. It is well suited for background planning, non-mutating suggestions, geometric search, candidate generation, or headless replay/validation. It is **not** the best owner of a live interactive editing session, because step execution is inherently stateful: Python variables, checkpoints, stale-handle generations, preview frames, and cell history all need affinity to one session. PythonŌĆÖs own executor docs also make clear that recycled worker processes and task limits are operational concerns in their own right; pools are valuable, but they should stay an implementation detail for background jobs, not the primary identity of an interactive Agent session. ŅłĆciteŅłéturn36view1Ņłü

The best fit is a **per-session Python subprocess**. One editing session gets one Python worker, one stateful namespace, one cancel token, one operation journal, one preview scene binding, and one lifecycle. That model gives you hard-kill recovery, easy timeout enforcement, per-session reproducibility metadata, clean teardown on cancel/reject, and a natural way to preserve notebook-like state between script cells. KiCad already launches IPC plugins in separate processes, and VS CodeŌĆÖs extension host exists for the same core reason: isolate extensibility from the editorŌĆÖs critical path. ŅłĆciteŅłéturn1view2Ņłéturn33view0Ņłü

On the wire protocol, **JSON-RPC over stdio** is the easiest thing to build, but it is not the best long-term fit. JSON-RPC is intentionally simple and transport-agnostic, and LSP proves it can scale to rich tooling ecosystems. But your use case has heavier needs than LSP: typed geometry payloads, atomically structured mutation records, step streaming, cancel/control messages that must cut past long-running work, and large preview artifacts that should usually move by handle rather than inline JSON blobs. JSON-RPC is excellent as a debug surface or compatibility bridge; it is weak as the permanent substrate of an AI-native CAD editor core. ŅłĆciteŅłéturn33view1Ņłéturn33view4Ņłü

**gRPC** is powerful and gives you bidirectional streaming and protobuf-based schemas, but it brings an HTTP/2 stack, more packaging complexity, more threading behavior, and more operational surface area than a same-machine GUI product usually needs. For cloud services, that complexity often pays for itself. For a local editor process talking to a local Python worker, it usually does not. KiCadŌĆÖs own public API chose **protobuf plus NNG/local IPC**, not gRPC, which is a strong precedent. ŅłĆciteŅłéturn33view3Ņłéturn1view2Ņłü

The best long-term choice is therefore **protobuf messages over local IPC**, with one **request/reply control channel** and one **event/observation stream**. If you want maximum alignment with KiCadŌĆÖs existing direction, build it on **NNG** and reuse the same transport family that KiCad already documents. NNG natively supports IPC transports and multiple messaging patterns including request/reply and pub/sub, while protobuf gives you the versioned typed schema discipline you need for a stable boundary. ProtobufŌĆÖs own best-practice guidance is directly relevant here: avoid breaking field changes, reserve deleted tags, include unspecified enum values, and keep RPC/API messages distinct from storage messages. ŅłĆciteŅłéturn1view2Ņłéturn35view0Ņłéturn35view1Ņłü

A very practical pattern is to borrow **JupyterŌĆÖs channel split** even if you do not adopt its full protocol: one shell/control path for requests and replies, one broadcast/event path for stdout, progress, preview-ready notifications, validation results, and debug events, plus a separate control/cancel path so a long-running execution cannot starve shutdown or interrupt handling. JupyterŌĆÖs documentation explicitly uses separate Shell, IOPub, and Control channels for exactly these reasons. ŅłĆciteŅłéturn33view2Ņłü

## Recommended session API shape and execution model

The public API that Python sees should be **Pythonic**, but the actual core contract should be a **typed session service**. In other words, Python code should say:

```python
with session.step("place stitching vias") as st:
    ring = st.create_zone(...)
    vias = []
    for i in range(n):
        vias.append(st.create_via(..., alias=f"via_{i}"))
obs = session.observe()
```

But under the hood, every mutating call should become a **typed atomic operation** recorded by C++ in the session journal: `CreateVia`, `CreateTrack`, `CreateZone`, `UpdateItem`, `DeleteItem`, `SetNet`, `SetPosition`, `RefillZones`, `RunValidation`, and so on. Python remains ergonomically first-class; the journal remains architecturally first-class. This mirrors KiCadŌĆÖs own push toward stable APIs rather than raw bindings, and it is exactly how you avoid coupling the editorŌĆÖs correctness model to Python object graphs. ŅłĆciteŅłéturn1view1Ņłéturn1view3Ņłéturn35view1Ņłü

The session API itself should be organized around a small number of calls with clear responsibility boundaries: `open_session`, `run_cell`, `begin_step`, `end_step`, `query_board`, `query_selection`, `query_viewport`, `query_activity`, `render_preview`, `run_checks`, `checkpoint`, `rollback_to`, `accept`, `reject`, and `cancel`. The Python SDK can collapse some of these into context managers and helper methods, but the wire contract should stay explicit. KiCadŌĆÖs own board API already exposes the precedent of typed board queries, item creation/update, selection access, and commit operations through a stable API surface rather than raw object mutation. ŅłĆciteŅłéturn34view0Ņłü

For execution style, the best fit is **stateful notebook/cell execution**, not one-shot script files and not generator-only semantics. The reason is not syntactic preference; it is the editing workflow you described. Agents need to run a bit of geometry, inspect the result, rollback, patch a parameter, and continue. JupyterŌĆÖs execution model is a very close analogy here: requests execute cells over a shell channel, side effects and progress stream over IOPub, and control messages can bypass long-running work. That pattern maps cleanly to AI-driven PCB construction. ŅłĆciteŅłéturn33view2Ņłü

Within a cell, the main bulk operation should be the **step**. A step is the unit that gets a preview refresh, validation pass, and automatic checkpoint. That means `with session.step("..."):` is not just a convenienceŌĆöit is your semantic boundary for progressive construction. This design also has strong precedent in CAD systems: FreeCAD exposes `openTransaction`, `commitTransaction`, and `abortTransaction`, while AutoCAD explicitly supports nested transactions where inner scopes can be committed or aborted independently before the outer transaction is finalized. ŅłĆciteŅłéturn22view0Ņłéturn38view0Ņłü

A good default rule is: **every step starts with an automatic checkpoint and ends with automatic preview, validation, and observation emission**. If Python raises an exception inside the step, or if core validation returns a configured ŌĆ£must-abortŌĆØ class of failure, the step should automatically rollback to its start checkpoint. If the step succeeds, the preview stays visible and the worker receives a `StepObservation` containing summary diff, created/updated handle aliases, warnings, frame handles, and validation results. That gives the Agent the same corrective loop that a human engineer uses. The comparison point from Fusion 360 is instructive: FusionŌĆÖs command API explicitly distinguishes preview from final execution and even lets the system mark the preview result as reusable when the user accepts. ŅłĆciteŅłéturn37view0Ņłéturn22view0Ņłéturn38view0Ņłü

Opaque object references are essential. Python should **never** hold raw `BOARD*` or `BOARD_ITEM*`, and ideally not even naked KiCad UUIDs as authority objects. Instead, return **session handles** that encode session id, revision generation, and underlying item identity. After rollback, any handle created in a reverted generation becomes stale. If Python tries to use it, the core should return a typed `STALE_HANDLE` error with enough metadata for the SDK to help the caller re-resolve the intended objectŌĆöby alias, user metadata, or current query. That may feel stricter than ordinary CAD scripting, but it is exactly what makes rollback, replay, and audit reliable in an Agent loop. KiCadŌĆÖs public APIs already distinguish persistent item identifiers such as KIID from live editor operations; your session layer should go one step further and make **generation validity** explicit. ŅłĆciteŅłéturn34view0Ņłéturn26search0Ņłü

## Preview, observation transport, and user-visible progressive execution

The worker should not receive raw board mutations as its primary feedback channel. It should receive a structured **observation object** after each step, with three layers of fidelity. The first layer is **structured state**: diff summary, created/modified/deleted items, bounding boxes, selected objects, active tool, viewport, validation findings, and maybe simplified geometry samples. The second layer is **semantic render metadata**: changed regions, highlighted items, scene categories, visibility settings, and frame descriptors. The third and most expensive layer is **pixels**: preview images rendered from the same native scene the user is seeing. This layering minimizes bandwidth and keeps the Agent from paying full image costs when structured state is enough. The pattern is conceptually similar to JupyterŌĆÖs separation of execution results from side-effect streams. ŅłĆciteŅłéturn33view2Ņłü

For visual transport, `render_preview` should usually return a **frame handle**, not image bytes inline. The handle can point to shared memory, a memory-mapped file, or a local blob store managed by the editor process. The request should support **region crop**, **viewport-aligned render**, **layer mask**, and optional semantic masks such as item-id or step-id channels. Large frame data should move out of band; the control reply should carry metadata only. This is an engineering recommendation rather than a documented KiCad API feature, but it is strongly supported by the fact that KiCadŌĆÖs public IPC is already out-of-process and typed, and by the fact that local IPC transports like NNG are best used for commands and notifications rather than repeatedly shoving full-resolution images through monolithic request bodies. ŅłĆciteŅłéturn1view2Ņłéturn35view0Ņłü

To keep Agent vision aligned with user vision, the renders should be produced by the **same KIGFX scene configuration** driving the native PCB canvas, or by an offscreen renderer using the same painter, layer ordering, and overlay configuration. KiCadŌĆÖs own `WIDGET_DIFF_CANVAS` is an important precedent: it builds a `KIGFX::VIEW`, attaches a painter, creates a `VIEW_GROUP`, places it on `LAYER_GP_OVERLAY`, makes an overlay with `MakeOverlay()`, and then updates item visibility and highlight state without mutating source items. That is almost exactly the composition mechanism an AI preview scene needs. ŅłĆciteŅłéturn14view1Ņłéturn14view2Ņłéturn29search0Ņłü

This also solves the ŌĆ£user watches the AI workŌĆØ requirement. Your UI does not need to wait for final accept to show meaningful progress. Each step can land in the **AI preview layer**, with hoverable metadata and activity-timeline linkage, while remaining outside the real board model. KiCad already uses preview items and overlay-like mechanisms across tools and panelsŌĆöselection areas, ruler items, diff overlays, phase-skew overlays, and temporary inspection graphics all demonstrate that the editor already has a notion of temporary, explanatory, or analytical drawing separate from durable board content. ŅłĆciteŅłéturn30search4Ņłéturn30search5Ņłéturn30search7Ņłü

A good observation event model is therefore:

- `step_started`
- `stdout` / `stderr`
- `preview_updated`
- `validation_completed`
- `frame_ready`
- `checkpoint_created`
- `rollback_completed`
- `step_failed`
- `session_stale`
- `accept_completed`

That event split is not arbitrary; it mirrors what editors and notebook systems do when they need long-running tools to remain inspectable and cancellable while still feeling interactive. ŅłĆciteŅłéturn33view0Ņłéturn33view2Ņłü

## Cancellation, determinism, replay, and final accept

Because KiCad is fundamentally a single-threaded GUI application at the editing surface, your communication model must support both **soft cancel** and **hard kill**. Soft cancel means sending a control message that sets a session cancel token and asks cooperative code to stop at safe pointsŌĆöbetween Python bytecode callbacks to session API, between steps, or between validator subphases. Hard kill means terminating the Python subprocess if it wedges or spins. PythonŌĆÖs standard library explicitly supports subprocess timeouts and kill/wait handling; that is much harder to do robustly if the interpreter is embedded in the GUI process. ŅłĆciteŅłéturn1view2Ņłéturn36view0Ņłü

The control path should be physically separate from the main request path. JupyterŌĆÖs docs make the reason explicit: shutdown and control messages should not queue behind long-running execution work. The same principle applies here. If a Python worker is busy computing a bad geometry loop, the editor must still be able to interrupt it, collect its partial output, tear down preview state, and return the GUI to a coherent state. ŅłĆciteŅłéturn33view2Ņłü

For determinism and audit, record at least: **session id, document specifier, base board hash, editor build/version, Python version, package environment fingerprint, cell history, explicit random seed, session options, checkpoints, and the full typed operation journal**. Protobuf best practices are useful here because they remind you to treat the journal schema like a real evolving protocol: reserve deleted fields, keep enums extensible, and avoid overloading interchange messages with storage concerns. The key design principle is that replay should be based on **journal plus initial snapshot**, not on ŌĆ£hope the Python code still behaves the same.ŌĆØ ŅłĆciteŅłéturn35view1Ņłü

This is also why Python should have restricted ambient capabilities during Agent execution even if you are not optimizing for adversarial sandboxing. For reproducibility, `time`, `random`, filesystem reads, network calls, and unmanaged threads should be **denied by default or wrapped**. If you allow them, the run should be explicitly tagged as **non-reproducible**. In the same way that CAD APIs and notebook systems separate deterministic model edits from arbitrary side effects, your session runtime should separate **replayable board construction** from **best-effort external behavior**. That recommendation is an inference from the replay requirements, but it is strongly aligned with the protocol-and-host separation principles used by KiCad IPC, VS Code extension hosts, and Jupyter kernels. ŅłĆciteŅłéturn1view2Ņłéturn33view0Ņłéturn33view2Ņłü

Accept should be a **core-owned promotion**, never a Python-owned direct write. The workerŌĆÖs job ends at ŌĆ£I have produced a validated preview journal on top of a known base board state.ŌĆØ The coreŌĆÖs accept job is then either to **replay the journal into the live board under one commit** or to **promote a shadow-board diff under one commit**, after first checking that the live board still matches the previewŌĆÖs base hash. If it does not, the preview is stale and must be rebased or rejected. KiCadŌĆÖs current APIs already group changes into a single commit/undo step, and `BOARD_COMMIT` exists in the native source precisely as the staging/push/revert mechanism for this territory. ŅłĆciteŅłéturn34view0Ņłéturn6view3Ņłü

The simplest first implementation is to replay the typed journal into the live board under a single `BOARD_COMMIT`, because that keeps the accept path aligned with the same atomic ops your audit/replay system already understands. A later optimization can switch parts of accept to diff promotion if benchmarking shows that journal replay is too slow for very large preview sessions. But even then, keep the journal as the canonical explanation layer; diffs alone are poor forensic artifacts compared with typed operation logs. AutoCADŌĆÖs transaction model and FreeCADŌĆÖs command transactions both reinforce the same general lesson: the stable user-facing semantics come from transaction boundaries, not from whatever internal object mutation sequence happened to realize them. ŅłĆciteŅłéturn22view0Ņłéturn22view1Ņłéturn22view4Ņłéturn38view0Ņłü

## Recommended architecture and the first shippable MVP

The recommended long-term architecture is:

- **C++ KiSurf core inside the editor process** owns live board access, shadow session/board, typed atomic operation execution, preview scene generation, validation, checkpoints, rollback, and final accept into the native undo stack. ŅłĆciteŅłéturn1view2Ņłéturn26search0Ņłéturn27search4Ņłéturn6view3Ņłü
- **Python runs out of process, one worker per interactive session**, with a persistent namespace across cells. It only sees a **Pythonic session SDK**, never raw KiCad internals. ŅłĆciteŅłéturn1view2Ņłéturn33view0Ņłü
- **Transport is protobuf over local IPC**, ideally NNG to match KiCadŌĆÖs existing IPC design and to support request/reply plus event streaming cleanly. ŅłĆciteŅłéturn1view2Ņłéturn35view0Ņłéturn35view1Ņłü
- **Execution model is stateful cells plus explicit step scopes**. Each step creates an automatic checkpoint, emits preview and validation on close, and can be rolled back independently. ŅłĆciteŅłéturn33view2Ņłéturn22view0Ņłéturn38view0Ņłü
- **All mutations are typed atomic ops** recorded in a session journal. Python can do loops and geometry, but it cannot bypass the journal. ŅłĆciteŅłéturn1view1Ņłéturn35view1Ņłü
- **Preview is native-canvas overlay**, not direct mutation of the live board. The user sees the AIŌĆÖs progressive work in the same editor view; the final accept is a separate promotion step. ŅłĆciteŅłéturn14view1Ņłéturn29search0Ņłü

The first shippable MVP should be intentionally narrow. Implement a per-session Python subprocess, a protobuf session service, a Python SDK exposing `session.step`, `create_via`, `create_track`, `create_zone`, `query_board`, `query_selection`, `query_viewport`, `render_preview`, `run_checks`, `rollback_to`, `accept`, `reject`, and `cancel`, an in-editor AI preview overlay layer, automatic per-step checkpoints, and final promotion under one `BOARD_COMMIT`. For the medium term, add prewarmed workers for faster startup, background read-only Preview Agent sessions, richer validation stratification such as geometry checks first and full DRC later, frame-handle transport with region crops, and a rebase path for stale previews when the live board changes during a long session. KiCadŌĆÖs public IPC roadmap and current `kicad-python` evolution show that extending stable typed APIs over time is feasible as long as the protocol boundary stays disciplined. ŅłĆciteŅłéturn1view1Ņłéturn32view0Ņłéturn34view0Ņłéturn35view1Ņłü

For the long term, keep Python as the **default and primary** scripting language if that is best for your users and agentsŌĆöbut do **not** let Python become the only architectural boundary. The boundary that must endure is the **typed KiSurf session protocol plus operation journal**. If that boundary is stable, you can add JavaScript, Lua, WASM, or model-native planners later without rewriting the core. If that boundary is unstable, then even a Python-only strategy will eventually force a rewrite. That is the central architectural conclusion from KiCadŌĆÖs shift away from raw SWIG bindings, from the LSP/extension-host pattern in editors, and from transaction/preview patterns across CAD systems.
